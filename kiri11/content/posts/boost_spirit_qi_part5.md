<h3><strong><code>BOOST_FUSION_ADAPT_STRUCT</code> — автоматическое распространение атрибутов для пользовательских типов.</strong></h3>
В предыдущих частях (<a title="часть первая" href="http://kiri11.ru/boost_spirit_qi_part1/">1-я</a>, <a title="часть вторая" href="http://kiri11.ru/boost_spirit_qi_part2/">2-я</a>, <a title="часть третья" href="http://kiri11.ru/boost_spirit_qi_part3/">3-я</a>, <a title="часть четвёртая" href="http://kiri11.ru/boost_spirit_qi_part4/">4-я</a>) я говорил об автоматическом заполнении STL структур атрибутами при парсинге. Таким образом мы, например, заполняли список пар "ключ=значение", пользуясь тем, что сами пары могли автоматически конвертироваться в <code>std::pair&lt;std::string, std::string&gt;</code>, а список пар спокойно мог заполнить собой <code>std::vector&lt;std::pair&lt;std::string, std::string&gt; &gt;</code>. Весь этот механизм, как тоже было сказано ранее, основывается на том, что для стандартных контейнеров STL существуют специальные обёртки, благодаря которым они могут считаться полноценными <code>boost::fusion</code> последовательностями.

Как быть, если мы хотим, чтобы наша собственная структура при парсинге могла бы автоматически заполняться так же легко, как это например делают <code>std::vector</code> или <code>std::string</code>? Оказывается, способ есть. Притом достаточно простой. Я не зря сказал про обёртки, которыми снабжены стандартные контейнеры библиотеки STL. В нашем случае придётся сделать в точности то же самое — снабдить нашу структуру всем необходимым в этом плане, иными словами: сказать Boost.Fusion, что наша структура является полноправным Fusion контейнером, построенном на модели <code>random access sequence</code>. Специально для этого в Fusion есть макрос <code>BOOST_FUSION_ADAPT_STRUCT</code>, который позволяет адаптировать любую пользовательскую структуру в полноправный Fusion контейнер.<!--more-->

Выглядит он примерно следующим образом:

```cpp
BOOST_FUSION_ADAPT_STRUCT(
    struct_name,
    (member_type0, member_name0)
    (member_type1, member_name1)
    ...
    )
```

Макрос должен использоваться в глобальном пространстве имён. Теперь о том, что здесь к чему: <code>struct_name</code> — полностью квалифицированное имя структуры (со всеми пространствами имён), которую хотим адаптировать. Пары <code>(member_typeN, member_nameN)</code> определяют типы и имена каждого члена структуры, которые станут частью адаптированной последовательности.

Пример:

```cpp
namespace Example
    {
    struct ExampleStruct
        {
        int IntField;
        std::string StringField;
        };
    }

BOOST_FUSION_ADAPT_STRUCT
    (
    Example::ExampleStruct,
    (int, IntField)
    (std::string, StringField)
    )
```

Для того, чтобы пользоваться этим добром, нужно, чтобы был подключён следующий заголовочный файл:

```text

```

И напоследок ещё один примерчик, выдранный из документации к Спириту. Нужно заполнять структуру данных, описывающую рабочего с такими параметрами как возраст, имя, фамилия и зарплата. Для этого пишем соответствующую структуру:

```cpp
struct Employee
    {
    int Age;
    std::string Surname;
    std::string Forename;
    double Salary;
    };
```

Теперь делаем из этой структуры Fusion контейнер с помощью макроса <code>BOOST_FUSION_ADAPT_STRUCT</code>:

```cpp
BOOST_FUSION_ADAPT_STRUCT
    (
    Employee,
    (int, Age)
    (std::string, Surname)
    (std::string, Forename)
    (double, Salary)
    )
```

Теперь нужно подумать о том, в каком виде записи о рабочих надо парсить. Пускай будет примерно так: <code>employee{ age, "surname", "forename", salary }</code>. Пишем соответствующий парсер:

```cpp
template 
struct EmployeeParser
    : qi::grammar
    {
    EmployeeParser()
        : EmployeeParser::base_type(employee_rule)
        {
        quoted_string = qi::lexeme['"' >> +(qi::char_ - '"') >> '"'];

        employee_rule =
                qi::lit("employee")
            >>  '{'
            >>  qi::int_ >> ','
            >>  quoted_string >> ','
            >>  quoted_string >> ','
            >>  qi::double_
            >> '}'
            ;
        }

    qi::rule quoted_string;
    qi::rule employee_rule;
    };
```

Реальный тип атрибута от выражения правой части правила <code>employee_rule</code> является следующим:

```cpp
fusion::vector
```

Поскольку мы адаптировали структуру <code>Employee</code>, чтобы она была нормальной Fusion последовательностью, то приведённый выше тип <code>fusion::vector&lt;...&gt;</code> может быть сконвертирован в тип структуры <code>Employee</code> точно так же, как это могут делать стандартные контейнеры STL, типа <code>std::pair</code>, <code>std::map</code> или <code>std::vector</code>, что позволяет полагаться на автоматическое распространение атрибутов без необходимости писать семантические действия в правиле, направленные на заполнение необходимых полей данной структуры.
<h3><strong>Удобная работа с ключевыми словами — директива <code>kwd[]</code>.</strong></h3>
В Спирите есть очень удобный механизм для распознавания входных данных, основанных на ключевых словах — директивы <code>kwd[]</code> и <code>ikwd[]</code>. Для того, чтобы работать с ними, должен быть подключён заголовок

```text

```

Находится директива в пространстве имён <code>boost::spirit::repository::qi</code>. Синтаксис директивы <code>kwd[]</code> может быть одним из следующих:

```text
kwd(keyword)[subject]
kwd(keyword, n)[subject]
kwd(keyword, min, max)[subject]
kwd(keyword, min, inf)[subject]
```

Теперь, собственно, о том, для чего каждый вариант служит:
<ol>
	<li><code>kwd(keyword)[subject]</code>
Этот вариант парсит такие строки: <code>("keyword" &gt; subject)</code> нуль и более раз.</li>
	<li><code>kwd(keyword, n)[subject]</code>
Такая штука парсит <code>("keyword" &gt; subject)</code> ровно <code>n</code> раз.</li>
	<li><code>kwd(keyword, min, max)[subject]</code>
Парсинг выражения <code>("keyword" &gt; subject)</code> от <code>min</code> до <code>max</code> раз.</li>
	<li><code>kwd(keyword, min, inf)[subject]</code>
Обрабатывает выражение <code>("keyword" &gt; subject)</code> как минимум <code>min</code> раз.</li>
</ol>
В начале я упомянул про директиву <code>ikwd[]</code> — она отличается от <code>kwd[]</code> только тем, что ключевые слова будут распознаваться без учёта регистра символов. Смешивать вместе <code>kwd[]</code> и <code>ikwd[]</code> можно, но всё-таки не рекомендуется, если что.

Теперь по поводу атрибутов от этой директивы: атрибут части, отвечающей за ключевое слово, игнорируется, так что распространяется атрибут только от парсера <code>subject</code>, который преобразуется согласно обычным правилам превращения атрибутов при секвенциальных конструкциях и в общем случае может быть представлен в виде STL-совместимого контейнера. На всякий случай приведу небольшую справочку по этому поводу:

```cpp
///////////////////////////////////////
// kwd(k1)[a]
a: A --> kwd(k1)[a]: optional<a> or vector</a><a> a: Unused --> kwd(k1)[a]: Unused /////////////////////////////////////// // kwd(k1,n)[a] a: A --> kwd(k1,n)[a]: optional</a><a> or vector</a><a> a: Unused --> kwd(k1,n)[a]: Unused /////////////////////////////////////// // kwd(k1,min, max)[a] a: A --> kwd(k1,min, max)[a]: optional</a><a> or vector</a><a> a: Unused --> kwd(k1,min, max)[a]: Unused /////////////////////////////////////// // kwd(k1,min, inf)[a] a: A --> kwd(k1,min, inf)[a]: optional</a><a> or vector</a><a> a: Unused --> kwd(k1,min, inf)[a]: Unused</a>
```

Как быть, если нужно парсить связку из данных, разделённых разными ключевыми словами?
Для этого есть специальный оператор <code>'/'</code>, применяемый с директивами <code>kwd[]</code> и <code>ikwd[]</code>. Например, запись <code>kwd("k1")[a] / kwd("k2")[b]</code> будет эквивалентна следующему коду:

```text
*("k1" > a | "k2" > b)
```

То бишь, этот оператор очень схож с оператором выбора альтернативы в некотором смысле. Как только распознано одно из ключевых слов, входящих в связку, сразу же будет сделана попытка применить парсер <code>subject</code>, стоящего за данным ключевым словом. Можно, конечно, и проигнорировать существование данной директивы и писать напрямую через оператор альтернативы, но тогда мы потеряем в скорости, ибо при связке нескольких директив с помощью оператора <code>'/'</code> внутри этого всего дела конструируется <a title="ternary search tree" href="http://en.wikipedia.org/wiki/Ternary_search_tree">троичное дерево поиска</a> для эффективного парсинга ключевых слов, да и удобней так, в конце концов, чего писать велосипеды, когда есть такая фича уже "из коробки".

Теперь разберём маленький пример на эту тему.
Предположим, что нам нужно распарсить запись о человеке, которая включает в себя его имя, возраст и его любимые цвета.
Данные представлены следующей структурой данных:

```cpp
struct Person
    {
    std::string name;
    unsigned age;
    std::vector favourite_colours;
    };

BOOST_FUSION_ADAPT_STRUCT
    (
    person,
    (std::string, name)
    (unsigned, age)
    (std::vector, favourite_colours)
    )
```

Ок, теперь о том, как принимать данные в записи: возраст и имя у человека может быть только одно формально, так что встретить соответствующие ключевые слова мы должны ровно один раз. Количество любимых цветов, напротив, может быть каким угодно, в том числе таковых просто может не оказаться.

На основе этих данных пишем правило для заполнения структуры <code>Person</code>:

```cpp
person %=
      kwd("name", 1)         [ '=' > string_rule ]
    / kwd("age" , 1)         [ '=' > int_        ]
    / kwd("favourite colour")[ '=' > string_rule ]
    ;
```

<code>string_rule</code> здесь выступает в качестве некоторого правила, распознающего строки, например, заключённые в кавычки — так удобней.

Теперь мы имеем возможность парсить строки примерно следующего вида прямо в структуру <code>Person</code>:

```cpp
"name = \"John\" \\n age = 10 "
"age = 10 \\n name = \"Mary\""
"name = \"Hellen\" \\n age = 10 \\n favorite colour = \"blue\" \\n favorite colour = \"green\" "
```

Заметьте то, что порядок следования ключевых слов значения не имеет, записи могут присутствовать в любом порядке.

Между тем, следующая запись провалит парсинг, потому что в ней не соблюдены условия на количество появлений ключевого слова <code>"age"</code>:

```cpp
"name = \"Johny\"  \\n favorite colour = \"blue\" \\n favorite colour = \"green\" "
```

<h3><strong>Таблицы символов <code>qi::symbols</code>.</strong></h3>
Класс <code>symbols</code> представляет из себя ассоциативный контейнер пар <code>ключ-значение</code>, где ключи представляются в виде строк, а тип значения выбирается в соответствии со вторым шаблонным параметром. Этот класс может нормально работать со строками <code>8</code>, <code>16</code>, <code>32</code>, <code>64</code>-битного символьного типа. Класс <code>symbols</code> является примером "динамического" парсера — то есть парсера, поведение которого может быть изменено в ходе выполнения программы. Чтобы пользоваться этим парсером, нужно подключить следующий заголовок:

```text

```

Заголовок этого шаблонного класса выглядит следующим образом:

```cpp
template 
struct symbols;
```

<code>Char</code> описывает тот символьный тип, который будет использоваться для ключевых строк в таблице символов. <code>T</code> — тип данных, ассоциированный с каждой записью в таблице, <code>Lookup</code> отвечает за тип внутреннего представления структуры поиска ключей в таблице.

Интерфейс для добавления, удаления или присвоения записей в таблицы <code>qi::symbols</code> в некотором смысле схож с тем, как реализованы подобные вещи в библиотеке <code>boost::assign</code>. Так, для присвоения таблице некоторого набора <code>(s1, s2, ..., sN)</code> символов можно воспользоваться нотацией

```cpp
sym = s1, s2, s3, ..., sN;
```

Для добавления символов в таблицу можно воспользоваться одним из следующих вариантов:

```cpp
// add symbols to qi::symbols instance via operator+=
sym += s1, s2, s3, ..., sN;
// add symbols via 'add' adapter
sym.add(s1)(s2)...(sN);
// add symbols (s1, ..., sN) with associated data (d1, ..., dN)
sym.add(s1, d1)(s2, d2)...(sN, dN);
```

Заметьте, в случае использования <code>add</code> мы имеем возможность как указывать каждой записи ассоциированное значение, так и не указывать его.

Для удаления символов из таблицы используются похожие функции, называются <code>operator-=</code> и <code>remove</code> соответственно:

```cpp
sym -= s1, ..., sN;
sym.remove(s1)(s2)...(sN);
```

Для очистки таблицы есть функция <code>clear</code>, используется так же, как и в STL контейнерах.

Внутреннее представление данных может быть различно в зависимости от того, какой вариант реализации выбираем, а их на данный момент две штуки: либо всё те же троичные деревья поиска (TST), что и в реализации директив <code>kwd[]</code>, либо гибридный вариант из hash-map для первого символа плюс TST, что иногда бывает существенно быстрее, нежели классическая реализация только на TST, только по затратам памяти несколько больше выходит. Настроиться на использование второго варианта можно, указав в качестве третьего шаблонного параметра для класса <code>symbols</code> тип <code>tst_map</code>:

```cpp
symbols > sym;
```

Небольшой примерчик:

```cpp
qi::symbols sym;

sym.add("fuck", 0)
    ("suck", 1)
    ("shit", 2)
    ("bullshit", 3)
    ;

int myChoice;
test_parser_attr("fuck", sym, myChoice);
std::cout << myChoice << std::endl; // prints "0"
```

Также довольно часто наследуются от <code>qi::symbols</code> и внутри конструктора описывают добавление необходимых записей в таблицу символов, в связи с этим можно привести пример, упоминающийся в официальной документации, хорошо иллюстрирующий пользу от использования парсера <code>symbols</code>: нужно распознавать римские числа, естественно с возможностью иметь вычисленное значение в эквиваленте наших арабских циферкок.

Для этого создадим несколько таблиц для представления единиц, десятков и сотен:

```cpp
struct ones_ : qi::symbols
    {
    ones_()
        {
        add
            ( "I"   , 1 )
            ( "II"  , 2 )
            ( "III" , 3 )
            ( "IV"  , 4 )
            ( "V"   , 5 )
            ( "VI"  , 6 )
            ( "VII" , 7 )
            ( "VIII", 8 )
            ( "IX"  , 9 )
            ;
        }
    } ones;

struct tens_ : qi::symbols
    {
    tens_()
        {
        add
            ( "X"   , 10 )
            ( "XX"  , 20 )
            ( "XXX" , 30 )
            ( "XL"  , 40 )
            ( "L"   , 50 )
            ( "LX"  , 60 )
            ( "LXX" , 70 )
            ( "LXXX", 80 )
            ( "XC"  , 90 )
            ;
        }
    } tens;

struct hundreds_ : qi::symbols
    {
    hundreds_()
        {
        add
            ( "C"   , 100 )
            ( "CC"  , 200 )
            ( "CCC" , 300 )
            ( "CD"  , 400 )
            ( "D"   , 500 )
            ( "DC"  , 600 )
            ( "DCC" , 700 )
            ( "DCCC", 800 )
            ( "CM"  , 900 )
        }
    } hundreds;
```

Для каждого элемента римских чисел мы имеем ассоциированное значение, теперь можно и основной парсер написать, чтобы распознавать произвольные числа в римской системе записи:

```cpp
template 
struct roman
    : qi::grammar
    {
    roman()
        : roman::base_type(StartRule)
        {
        using qi::eps;
        using qi::lit;
        using qi::_val;
        using qi::_1;
        using ascii::char_;

        StartRule = eps         [_val = 0] >>
            (
                +lit('M')       [_val += 1000]
                ||  hundreds    [_val += _1]
                ||  tens        [_val += _1]
                ||  ones        [_val += _1]
            )
            ;
        }

    qi::rule StartRule;
    };
```

По ходу дела встретились ещё с одним оператором парсинга в Спирите: секвенциальный условный оператор <code>'||'</code>. В выражении <code>a || b</code> этот оператор значит следующее: распознавать либо <code>a</code>, либо <code>b</code> в последовательности. То есть если и <code>a</code>, и <code>b</code> будут распознаны, то они должны идти по порядку. Возможно, объяснение не совсем хорошее, так что если формально, то действие этого оператора будет эквивалентно следующему выражению, только несколько более эффективно:

```cpp
a >> -b | b
```

Короче, если у вас есть много однотипных данных-токенов, да ещё и при условии, что есть вероятность того, что появятся новые объявления, то пользуйтесь парсером <code>symbols</code>, дабы не строить велосипеды через многочисленные операторы альтернатив с приаттаченными семантическими действиями, и будет вам счастье.
<h3><strong>Директива <code>distinct[]</code>.</strong></h3>
Директива <code>distinct[]</code> предназначена для того, чтобы избежать неполных случаев распознавания строк при использовании пропускателей (парсеров пропусков). Обычно применяется для того, чтобы безопасным образом отделять ключевые слова во входных данных. Рассмотрим следующий пример: допустим, нам нужно распознавать строчки вида <code>data[':']&lt;some_data&gt;</code>, то есть слово <code>data</code> после которого может либо стоять двоеточие, либо не стоять, а дальше необходимые данные до конца строки. Как можно решить данную задачу:

```cpp
"data" >> -lit(':') >> *(char_ - eol)
```

Вроде ништяк, но на самом деле есть проблемы: предположим, мы используем стандартный пропускатель <code>space</code>, тогда следующие три варианта одинаково успешно пройдут процедуру распознания:

```text
"data: some_data\\n"
"data some_data\\n"
"datasome_data\\n"
```

Немного неожиданно, конечно, но так и будет, а всё из-за того, что в виду возможности отсутствия двоеточия после <code>data</code> вступает в дело парсер <code>*(char_ - eol)</code>, который успешно съедает все символы в строке, потому как пропускатель ничего не может с этим поделать, даже если необходимых пробелов нет.

Чуть-чуть подумав, приходим к примерно такому решению проблемы:

```cpp
lexeme["data" >> !(alnum | '_')] >> -lit(":") >> *(char_ - eol)
```

Оператор <code>'!'</code> (называется обычно "not-предикат") предназначен для проверок хода парсинга, делает примерно следующее: если в выражении <code>!p</code> парсер <code>p</code> завершает работу успешно, то парсинг проваливается, и наоборот. Обратный вариант этого оператора — оператор <code>'&amp;'</code> ("and-предикат") — завершается успехом только тогда, когда подконтрольный парсер отрабатывает успешно. Отличительная черта этих парсеров в том, что они, точно так же, как и парсер <code>eps</code>, не поглощают входных данных, возвращая результат считывания нулевой длины. Атрибут подконтрольных парсеров тоже игнорируется, то есть, если <code>a</code> имеет атрибут типа <code>A</code>, то парсер <code>!a</code> имеет атрибут типа <code>unused_type</code>, то бишь он просто не используется.

В данном примере мы считываем ключевое слово <code>"data"</code>, а с помощью <code>not</code>-предиката удостовериваемся, что <em>непосредственно</em> после этого слова не идёт валидный идентификатор, который пойдет в качестве данных, ассоциированных с этим ключевым словом. Теперь из перечисленных вариантов входных данных только первые два отрабатывают успешно, что было нужно изначально.

Вот именно для таких случаев и применяют обычно директиву <code>distinct[]</code>, с её использованием предложенный чуть выше вариант будет выглядеть так:

```cpp
distinct(alnum | '_')["data"] >> -lit(":") >> *(char_ - eol)
```

Кроме того, что выглядит чуть попроще, как обычно это и бывает с директивами, также появляется потенциал для более удобного повторного использования и инкапсуляции хвостового парсера для разделения (в нашем случае это <code>alnum | '_'</code>) в отдельную конструкцию.

В официальной документации даже можно найти <a title="distinct parser directive" href="http://www.boost.org/doc/libs/1_44_0/libs/spirit/repository/doc/html/spirit_repository/qi_components/directives/distinct.html">примерчик</a> того, как приведённый пример можно ещё больше упростить за счет объявления синонима для директивы <code>distinct(alnum | '_')</code>, но это уже включает более глубокий взгляд во внутренности того, как работают такие механизмы внутри самой библиотеки, так что об этом я здесь говорить не буду, кому надо, тот почитает в документации с необходимыми пояснительными моментами.

Для использования подключаем следующий заголовок:

```text

```

Сама директива находится в пространстве имён <code>boost::spirit</code>, но для неё также существует синоним в пространстве имён <code>boost::spirit::qi</code>.

Указанный интервал символов, возвращаемый директивой, является объектом типа <code>boost::iterator_range&lt;Iterator&gt;</code>, который держит пару итераторов на начало и конец произвольного интервала с заданным типом итератора. Правила распространения атрибута следующие:

```cpp
///////////////////////////////
// raw[a] directive
a: A --> raw[a]: boost::iterator_range
a: Unused --> raw[a]: Unused
```

Данная директива имеет только одно основное применение, основанное на соображениях производительности: например, распознавая строку в парсере с атрибутом типа <code>std::string</code>, при успешном парсинге будет создан объект <code>std::string</code> с заданной строкой, а потом он будет ещё раз скопирован уже в ту строку, что мы, например, ожидаем при выходе из фунции <code>parse</code>. Применяя директиву <code>raw[]</code>, удаётся избежать ненужного копирования и создания объектов, которые в Boost.Spirit происходят очень-очень часто и с чём люди борются различными способами.

Типичный пример использования:

```cpp
std::string input = "fucking_shit";
std::string output;
parse(input.begin(), input.end(), raw[*(alpha | '_')], output);
std::cout << output << std::endl; // prints fucking_shit
```

<h3><strong>Частичное применение функций и <code>phoenix::bind</code>.</strong></h3>
В <a title="часть третья" href="http://kiri11.ru/boost_spirit_qi_part3/">третьей части</a> я рассказывал о том, как можно адаптировать обычные функции и функциональные объекты для исполнения в ленивой манере. Для этого мы пользовались услугами <code>phoenix::function</code> — класса, имитирующего ленивые функции. Но ленивые функции не отходя от кассы можно получать и несколько другим способом. В той же третьей части я упоминал термин "частичное применение функций", именно о нём и пойдёт речь. Это явление также принято называть связыванием (binding) — операция привязывания некоторых аргументов функции для последующего отложенного выполнения. В общем случае <code>phoenix::bind</code>, равно как и <code>boost::bind</code>, является обобщением стандартных функций <code>std::bind1st</code> и <code>std::bind2nd</code>, которые позволяли в качестве первого или второго аргумента некоторой функции привязать некоторое значение, получая на выходе уже унарный функтор, где вместо одного из аргументов находится привязанное значение, который можно удобно передавать в стандартные алгоритмы.

Впрочем, для тех, кто по каким бы то ни было причинам не пользовался указанными функторами <code>std::bind1st</code> или <code>std::bind2nd</code>, или нигде (хотя бы и в другом языке) не сталкивался вообще с понятием частичного применения функций, мои объяснения вряд ли внесут особую ясность, так что сейчас пойдут наглядные примеры, описывающие суть данного процесса.

С помощью <code>phoenix::bind</code> можно привязывать аргументы в произвольных функциях, функторах, функциях-членах классов, а так же можно привязывать поля данных в классах и структурах, короче, полный набор.

Например, пусть у нас есть следующая функция:

```cpp
void func(int num)
    {
    std::cout << num;
    }
```

Тогда её можно привязать для отложенного выполнения следующим образом:

```cpp
phoenix::bind(&f, phoenix::placeholders::_1)
```

Получившийся в результате этой операции объект является ленивым унарным функтором, который может быть использован аналогично ленивым функциям <code>phoenix::function</code>, например, так:

```cpp
// prints 10
bind(&f, _1)(10);
```

Также можно привязывать различным аргументам функции конкретные значения. Например, пусть даны функции

```cpp
int foo(int a, int b)
    {
    return a * b;
    }

int bar(int a, int b, int c)
    {
    return a * b + c;
    }
```

Тогда мы можем привязать функции <code>foo</code> в качестве первого аргумента, например, значение <code>1</code>: <code>bind(&amp;foo, 1, _1)</code> — результатом будет унарная ленивая функция, в качестве первого аргумента которой будет <code>1</code>. Вызов <code>bind(&amp;foo, 1, _1)(2)</code> будет эквивалентен вызову функции <code>foo(1, 2)</code>, что вернёт значение <code>2</code>. Приведённый только что пример с использованием стандартного связывателя <code>std::bind1st</code> выглядел бы так: <code>std::bind1st(std::ptr_fun(foo), 1)(2)</code>. Точно так же, выражение <code>bind(&amp;foo, _1, 1)(2)</code> привязывает второму аргументу фиксированное значение, что равносильно применению связывателя ("binder" — если по-буржуйски) <code>std::bind2nd</code>, вот то же действие с его использованием: <code>std::bind2nd(std::ptr_fun(foo), 1)(2)</code>. Вариант с <code>phoenix::bind</code>, равно как и <code>boost::bind</code> или <code>std::bind</code>, расширяет функциональность этих двух стандартных компонент связывания, добавляя возможность привязывать функции с произвольным числом аргументов.

С привязыванием аргументам фиксированного значения вроде бы всё достаточно просто, теперь чуть-чуть примеров, связанных с применением имён-заместителей <code>phoenix</code>:

```cpp
bind(foo, _1, _1)(x, y); // foo(x, x)
bind(foo, _2, _2)(x, y); // foo(y, y)
bind(foo, _1, _2)(x, y); // foo(x, y)
bind(foo, _2, _1)(x, y); // foo(y, x)
bind(bar, _1, _1, _1)(x, y, z); // bar(x, x, x)
```

Довольно круто! Также имеется возможность цепного последовательного связывания, например, возможна такая схема: первый раз из трёх параметров связываем один, остается бинарная функция, потом связываем ещё один, теперь это унарная функция, наконец, связываем оставшийся аргумент и на выходе получается нульарная функция, которую мы вызываем.

Первое, что приходит в голову, при желании реализовать приведённую схему — просто вложить вызовы bind один в другой, наподобие следующего:

```cpp
bind(bind(bind(bar, 2, _1, _2), 4, _1), 5)
```

Здесь мы ожидаем, что на выходе получится нульарная функция типа <code>bar(2, 4, 5)</code>, но такой подход не работает, потому как для того, чтобы каждый вызов <code>bind</code> с соответствующими аргументами-заместителями работал нормально, нужно, чтобы для каждого из них эти подстановочные аргументы <code>_1</code> и <code>_2</code> представлялись в своём определённом контексте. Иными словами, тот <code>_1</code>, что находится в самом глубоком вызове <code>bind</code> должен иметь иной смысл, чем аргумент <code>_1</code> в более внешнем вызове <code>bind</code>. На деле этого не происходит и все применения имён-заместителей здесь соотносятся в одну кучу с одинаковым контекстом. В итоге, из-за того, что для каждого последующего вызова <code>bind</code> не создаётся новое окружение и область видимости, применяемая к подстановочным аргументам, чтобы они могли адекватно менять своё смысловое значение в каждом связывании, получаем громадное количество ошибок компиляции, так как семантика применения заместителей <code>_1</code>, <code>_2</code> оказывается неверной. Вот если бы существовал способ для каждого связывания определить своё окружение, в рамках которого заместители работали бы как надо... Впрочем, такой способ существует.

Я уже упоминал, что в Phoenix существуют механизмы для контроля областей видимости и создания локальных переменных, сейчас самое время поговорить об этом. А чуть позже вернёмся к этому примеру последовательного связывания аргументов.

<h4><strong>Объявление локальных переменных — блоки <code>let</code>.</strong></h4>
Для того, чтобы объявлять локальные переменные, в Phoenix существует конструкция под названием <code>let</code> с использованием следующего синтаксиса:

```cpp
let(local_declarations)
    [
    let_body
    ]
```

В шапке <code>local_declarations</code> для <code>let</code> возможно разместить от <code>1</code> до <code>N</code> объявлений локальных переменных через запятую (<code>N == BOOST_PHOENIX_LOCAL_LIMIT</code>). Каждое из объявлений представляется в следующем виде:

```cpp
local_id = lambda_expression
```

<code>let_body</code> представляет из себя последовательность ленивых statement'ов, разделённых между собой запятыми, точно так же, как это происходит в теле ленивых аналогов для <code>if, for, while, do_while</code>.

Для того, чтобы пользоваться, подключаем следующий заголовок:

```text

```

Простой пример использования:

```cpp
let(_a = 1, _b = 2)
    [
    std::cout << _a + _b << std::endl
    ]
```

То есть ситуация примерно следующая: в шапке конструкции <code>let</code> объявляем переменные вместе с их начальными значениями, а в теле, заключённом в квадратные скобки, действия, в пределах которых данная переменная будет находиться в области видимости, чем-то похоже на <code>using statement</code> в <code>C#</code> в том плане, что в шапке тоже происходит инициализация переменной и за пределами фигурных скобок, ограничивающих <code>using</code> выражение, переменная выходит из области видимости.

Тип объявляемой локальной переменной выводится из типа лямбда-выражения в правой части присваивания при инициализации, при этом в общем случае тип выводится с сохранением передачи по ссылке. Например:

```cpp
let(_a = _1, _b = 0) ...
```

В данном примере фениксовый плейсхолдер <code>_1</code> будет передан по ссылке, так что в <code>_a</code> окажется ссылка на значение подстановочного аргумента <code>_1</code>, в то время, как <code>_b</code> будет иметь значение типа <code>int</code>. Небольшой пример, наглядно демонстрирующий эффект от этой фичи:

```cpp
int var = 0;

let(_a = _1)
    [
    std::cout << ++_a << std::endl
    ](var)
    ;

std::cout << var;
```

Выведется два раза значение <code>1</code>, потому что захват переменной <code>var</code> через аргумент-заместитель происходит по ссылке. В то же время, следующий код напечатает 1, а потом <code>0</code>, потому как <code>val</code> является <code>rvalue</code>, и аргумент передаётся по значению:

```cpp
int var = 0;

let(_a = val(_1))
    [
    std::cout << ++_a << std::endl
    ](var)
    ;

std::cout << var << std::endl;
```

Таким образом, те штуки, которые считаются <code>lvalue</code> (а к ним относятся заместители типа <code>_<em>N</em></code> и ссылки <code>ref</code>), передаются по ссылке, а те, что являются <code>rvalue</code> (то есть непосредственные значения и <code>val</code>, например), передаются по значению.
<h4><strong>Область видимости переменных.</strong></h4>
Здесь, в принципе, всё аналогично тому, что мы имеем в ситуации с настоящими локальными переменными: время жизни и видимость переменной определяется телом вмещающего блока, для фениксовых переменных этот блок — тело <code>let</code>. При этом блоки <code>let</code> могут быть вложенными. В этом случае, если во внутреннем блоке была объявлена переменная с тем же именем, что и во внешнем блоке, то эта переменная скроет ту, что находится во внешней области видимости.

Небольшой примерчик:

```cpp
let(_a = "Fuck you", _b = ", World!")
    [
    let(_a = "Hello")
        [
        // prints "Hello, World!"
        std::cout << _a << _b << std::endl
        ]
    ]
```

По поводу присваиваний в заголовке <code>let</code> имеется один существенный момент: правая часть каждого локального объявления в заголовке <code>let</code> не может использовать ни один из идентификаторов локальных переменных, встречающихся в левых частях данного заголовка блока <code>let</code>. То есть следующий пример не прокатит, потому как переменная <code>_a</code> на момент объявления в заголовке <code>let</code> ещё не числится в зоне видимости:

```cpp
// error - _a is not in scope yet!
let(_a = 1, b = _a)
    [
    ]
```

В то же время, к локальным переменным из внешнего окружения можно спокойно достучаться. Следующий пример демонстрирует захват переменной из внешней области видимости во внутреннем блоке <code>let</code>.

```cpp
let(_a = "Hell yeah, motherfucker!")
    [
    let(_a = 0, _b = _a)
        [
        // now _a is int = 0
        // _b refers to the outer _a
        std::cout << _b // prints "Hell yeah, motherfucker!"
        ]
    ]
```

<h4><strong><code>phoenix::lambda</code> — создание новых областей видимости.</strong></h4>
Рассмотрим типичный пример с передачей функторов в алгоритмы. В качестве алгоритма возьмём реализацию "ленивого" <code>for_each</code>, которую я приводил ранее, если кто забыл, выглядело это примерно следующим образом:

```cpp
struct for_each_impl
    {
    template 
    struct result
        {
        typedef void type;
        };

    template 
    void operator() (Container & c, Function f) const
        {
        std::for_each(c.begin(), c.end(), f);
        }
    };

phx::function const for_each = for_each_impl();
```

В чём здесь соль: <code>operator()</code> в качестве аргумента принимает произвольную функцию <code>f</code>, которая уже передаётся стандартному <code>std::for_each</code>; область видимости данной функции <code>f</code> ограничена рамками <code>operator()</code>, в то же время, уже внутри функции <code>std::for_each</code> эта функция будет находиться в новой области видимости вместе с новыми аргументами. При этом та новая область видимости никак не будет связана с более внешними областями видимости далее, чем область <code>operator()</code>.

Собственно, переходим к тому, что только что сказанное представляет из себя на практике. Предположим, что с помощью нашего <code>for_each</code> нужно вывести все элементы контейнера в <code>cout</code>, при этом использовать <code>phoenix</code>-лямбду в качестве функтора, осуществляющего вывод. Наверняка, сразу же захочется написать что-то вроде этого:

```cpp
for_each(_1, std::cout << _1)
```

Несмотря на то, что мы вроде бы понимаем, что <code>_1</code> в качестве первого аргумента <code>for_each</code> является заместителем для контейнера, который будет передаваться, а <code>_1</code> в выражении с <code>std::cout</code> отвечает за конкретный элемент в контейнере, на деле получается немного не то. А что именно получается: поскольку оба заместителя <code>_1</code> находятся в "одной" области видимости, то они имеют одинаковую семантику, то есть, запись

```cpp
std::vector vec{1, 2, 3};
for_each(_1, std::cout << _1 << ' ')(vec);
```

по сути будет означать следующее: в качестве <code>_1</code> для первого аргумента передать контейнер <code>vec</code>, и в качестве <code>_1</code> для второго аргумента тоже передать контейнер <code>vec</code>, хотя нам бы хотелось, чтобы <code>_1</code> в части с <code>std::cout</code> относился бы к тому, что будет передаваться этой лямбде уже внутри вызова <code>std::for_each</code>, который запрятан внутри нашего рукописного <code>for_each</code>. Так вот для того, чтобы для этого заместителя в лямбде предоставить новую область видимости и чтобы он уже указывал на то, что будет передаваться непосредственно лямбде, приобрёл новое значение, нужно воспользоваться такой фигнёй под названием <code>phoenix::lambda</code>.

Чтобы использовать <code>lambda</code> нужно подключить заголовок

Синтаксис <code>lambda</code> блока может быть одним из следующих:

```cpp
lambda
    [
    lambda_body
    ]
```


```cpp
lambda(local_declarations)
    [
    lambda_body

```

В первом варианте новых локальных переменных не объявляем, а во втором случае объявляем, точно так же, как и у <code>let</code>: предел на количество объявлений переменных равен <code>BOOST_PHOENIX_LOCAL_LIMIT</code>. Точно те же, как и в случае с <code>let</code>, накладывается ограничение на применение и область видимости локальных переменных: в правых частях объявлений в заголовке <code>lambda</code> не может находиться ни одного идентификатора локальной переменной, который появляется тут же в левых частях объявлений, то есть следующая запись будет ошибочной по причине того, что переменная <code>_a</code> ещё не находится в области видимости объявляемой переменной <code>_b</code>:

```cpp
lambda(_a = 1, _b = _a) // error - _a not in scope yet!
    [
    // ...
    ]
```

Как будет выглядеть правильная реализация недавнего примера с применением данной фичи:

```cpp
std::vector vec{1, 2, 3};
for_each(_1, lambda[std::cout << _1 << ' '])(vec);
```

Тогда как аргумент <code>_1</code> задействован на уровне нашего <code>for_each</code> и обозначает переданный в качестве параметра контейнер, тело лямбды, заключённое в <code>lambda</code> существует в совершенно новой области видимости. Внутри <code>lambda</code>, заместитель <code>_1</code> уже отождествлён с тем, что будет передано самой лямбде в качестве параметра, когда дело дойдёт до непосредственного вызова, а произойдет это уже внутри функции <code>std::for_each</code>, где лямбде будут передаваться элементы контейнера <code>std::vector</code>, который мы передавали в ленивый <code>for_each</code>.

Также имеется возможность захвата аргументов из внешних областей видимости, чего, например, блок <code>let</code> делать не умеет. Рассмотрим маленький примерчик на эту тему. Пусть дан двумерный массив чисел (например, представленный в виде <code>std::vector&lt;std::vector&lt;int&gt; &gt;</code>) и некторое фиксированное число. Требуется запихнуть в каждый одномерный подмассив в конец это число. Для этого нам может потребоваться ленивый вариант функции контейнера <code>push_back</code>, пример которого я также приводил ранее, если что, напомню его вид:

```cpp
struct push_back_impl
    {
    template 
    struct result
        {
        typedef void type;
        };

    template 
    void operator() (Container & c, Arg & arg) const
        {
        c.push_back(arg);
        }
    };

phx::function

```

Теперь решение данной задачи с помощью наших ленивых <code>for_each</code>, <code>push_back</code>, <code>lambda</code> и вышеупомянутой фичи с захватом внешних аргументов:

```cpp
for_each(_1, lambda(_a = _2)
                [
                push_back(_1, _a)
                ]
        )
```

<h4><strong>Последовательное связывание с помощью <code>phoenix::bind</code>.</strong></h4>
Теперь мы понимаем, как сделать так, чтобы подстановочные аргументы приобретали в зависимости от контекста нужный смысл — нужно оборачивать каждый последующий вызов <code>phoenix::bind</code> в тело блока <code>lambda</code>, чтобы предоставить новую область видимости для подстановочных аргументов и тем самым добиться желаемого результата. Итого, выход будет выглядеть так:

```cpp
std::cout <<
    bind(lambda[
            bind(lambda[
                    bind(bar, 2, _1, _2)
                ], 4, _1)
        ], 5)() << std::endl;
// prints "13"
```

Что тут происходит: сначала мы привязываем первый аргумент функции <code>bar</code> со значением <code>2</code>, потом привязываем второй аргумент со значением <code>4</code>, а в конце привязываем оставшийся аргумент со значением <code>5</code> и вызываем получившуюся нульарную функцию, которая вычисляет выражение <code>2 * 4 + 5 == 13</code>.

<h3><strong>Лексический анализатор Spirit.Lex.</strong></h3>
Ещё одной частью Boost.Spirit является Spirit.Lex — генератор лексических анализаторов, довольно удобных в использовании и обладающих притом серьёзными возможностями. В принципе, как мы уже многократно убеждались, использование отдельного лексического анализатора для Spirit.Qi абсолютно необязательно, но по мере усложнения грамматик, было бы естественно разбивать их на множество более мелких в соответствии со смыслом и выполняемыми функциями, часто повторяющиеся токены выделить в отдельные конструкции, чтобы использовать для обращения к ним только мнемонические имена, всё это, несомненно, облегчит сопровождение всей системы в целом.

Spirit.Lex представляет собой библиотеку для генерации лексических анализаторов, входящую в состав Boost.Spirit и спроектированную таким образом, чтобы интеграция с другими частями библиотеки Boost.Spirit была максимально прозрачной. В частности, после небольшого знакомства с моделями описания токенов и самих лексеров, вы наверняка заметите много общего между тем, как ведут себя парсеры в Spirit.Qi и лексеры Spirit.Lex. Как и от использования Spirit.Qi в качестве инструмента парсинга, от библиотеки лексеров Spirit.Lex примерно всё те же достоинства по сравнению с другими инструментами лексического анализа:
<ol>
	<li>Нет дополнительной стадии компиляции. Не нужно пользоваться сторонними инструментами, которые уже с помощью различных неудобных интерфейсов нужно пристыковывать к готовому коду, всё делается непосредственно в коде C++.</li>
	<li>Возможность строить динамические лексические анализаторы, поведение которых можно изменять прямо по ходу дела.</li>
	<li>Возможность прицеплять к объявлениям токенов семантические действия в той же манере, как это делается и в Spirit.Qi, в том числе возможно использование Boost.Phoenix для составления семантических действий.</li>
	<li>Очень удобная и простая интеграция с компонентой для парсинга Spirit.Qi, да и вообще понятные и хорошие интерфейсы для работы.</li>
</ol>
Теперь о том, чем в основном приходится пользоваться в этой библиотеке и как оно примерно работает.

Большую часть времени Spirit.Lex используют как генератор динамических лексеров, что означает то, что необходимые таблицы для разбора генерируются во время выполнения из объявлений токенов, написанных, кстати, на языке регулярных выражений. Но, наряду с динамической моделью, существует также возможность использовать статическую модель, получая на выходе <a title="deterministic finite-state automaton" href="http://en.wikipedia.org/wiki/Deterministic_finite-state_machine">ДКА</a> (<a title="детерминированный конечный автомат" href="http://ru.wikipedia.org/wiki/%D0%9A%D0%BE%D0%BD%D0%B5%D1%87%D0%BD%D1%8B%D0%B9_%D0%B0%D0%B2%D1%82%D0%BE%D0%BC%D0%B0%D1%82">на русском</a>), что очень похоже на то, как действует генератор Flex, создающий из описания токенов, представленных на специальном языке, необходимые таблицы и функции для того, чтобы это всё добро как-то использовать. Таким образом, вопрос эффективности работы Spirit.Lex не стоит: если не устраивает то, как работает динамическая модель, можно запросто поменять динамический лексер на статический (как это сделать, можно <a title="static lexer model" href="http://www.boost.org/doc/libs/1_47_0/libs/spirit/doc/html/spirit/lex/abstracts/lexer_static_model.html">почитать в документации</a>) и всё будет просто замечательно.

Пространство имён библиотеки: <code>boost::spirit::lex</code>. Также основу многих концепций была взята реализация из библиотеки <code><a title="lexertl library" href="http://www.benhanson.net/lexertl.html">lexertl</a></code>, поэтому много что находится в соответствующем пространстве имён <code>boost::spirit::lex::lexertl</code>.

Основным элементом лексического анализатора являются токены. В Spirit.Lex для описания токенов нужно иметь представление о двух вещах и уметь их различать: определение токена (класс <code>lex::token_def&lt;&gt;</code>) и сам класс, представляющий токен (класс <code>lex::lexertl::token&lt;&gt;</code>).

Объявления токенов служат для того, чтобы:
<ul>
	<li>Упростить определение токена с помощью регулярных выражений для распознания данного типа токена.</li>
	<li>Указать тип атрибута для значения токена.</li>
	<li>Присвоить собственный идентификатор для типа токена.</li>
	<li>Ассоциировать тип токена с определённым состоянием лексера.</li>
	<li>Ассоциировать некоторый код с типом токена, который будет исполнен каждый раз, как будет распознан очередной токен данного типа (семантические действия, иными словами).</li>
</ul>
Сами токены являются структурами данных, которые возвращают итераторы лексического анализатора. При разадресации итератора лексера возвращается ссылка на объект последнего распознанного токена. Токены в себе содержат по меньшей мере идентификатор токена, который даёт возможность говорить о том, какой тип данных был считан из входного потока, также у них может быть ассоциированное значение (иначе, атрибут) и, возможно, состояние лексера, в котором они были считаны, но это уже необязательно, главное, чтобы у токена был идентификатор, по которому его можно однозначно распознать.

В то время, как <code>token_def</code> определяет свойства какого-то определённого токена, то сам <code>token</code> должен определять свойства, общие для всех объявлений токенов, которые входят в состав лексического анализатора, потому как именно тип токена является шаблонным параметром для типа лексера. То есть в рамках одного лексера может быть несколько определений токенов совершенно различного вида: одни могут в качестве атрибута возвращать строки <code>std::string</code>, другие числа типа <code>int</code>, третьи могут вообще игнорировать свой атрибут, а между тем класс <code>token</code> должен поддерживать каждое из этих определений в отдельности и, естественно, все их вместе тоже должен хорошо переваривать. Поэтому в общем случае типом атрибута токена является знакомый нам <code>boost::variant</code>, позволяющий содержать значение одного из перечисленных в списке шаблонных параметров типов. Естественно, в этом <code>boost::variant</code> типе должны быть указаны все типы атрибутов, которые появляются в объявлениях токенов, иначе получим ошибки компиляции.

Для типа токена обычно используется шаблонный класс <code>lex::lexertl::token&lt;&gt;</code>. По-умолчанию значение атрибута токена, если не указано явно, будет представлять собой интервал считанных символов <code>boost::iterator_range</code>, помещённый в класс <code>boost::variant</code>. Прототип этого шаблона выглядит примерно следующим образом:

```cpp
template <
    typename Iterator = char const*,
    typename AttributeTypes = mpl::vector0<>,
    typename HasState = mpl::true_
>
struct lexertl_token;
```

Параметр <code>Iterator</code> — это тип итератора, используемый для доступа к входному потоку. <code>AttributeTypes</code> — либо <code>mpl</code> последовательность, содержащая все типы, встречающиеся в определениях токенов <code>token_def</code>, либо тип <code>omit</code>, если атрибут должен быть проигнорирован. <code>HasState</code> — константа, определяющая, будет ли храниться в токене информация о том, в каком состоянии лексера он был сгенерирован, изначально равен <code>mpl::true_</code> — обёртка для булевого значения <code>true</code> из библиотеки <code>boost::mpl</code>, что означает то, что все токены по-умолчанию содержат в себе состояние лексера. Обычно, в процессе конструирования, объект токена содержит в себе значение типа <code>boost::iterator_range</code>, только если не был указан тип <code>omit</code> в качестве типа атрибута.

С самими токенами вроде бы разобрались, теперь чуть-чуть об определениях токенов <code>token_def</code>. Прототип класса <code>lex::token_def</code> выглядит следующим образом:

```cpp
template<
    typename Attribute = unused_type,
    typename Char = char
>
class token_def;
```

Точно так же: <code>Attribute</code> отвечает за тип атрибута, который будет порождён при генерации токена, а <code>Char</code> — за тип значения для итераторов во входной поток символов. Как и для самих токенов, если не указан явно тип атрибута, по-умолчанию атрибутом будет считаться <code>boost::iterator_range</code>, представляющий собой пару итераторов, указывающих на диапазон символов, считанных при распознании токена. Если же указан тип <code>omit</code>, то определение токена не будет содержать в себе объекта <code>iterator_range</code>. Так что если следует игнорировать атрибут токена и притом действовать самым эффективным путём, то нужно явно указывать, что атрибут для данного токена отсутствует прописыванием типа <code>omit</code> в качестве типа атрибута, иначе хоть и будет работать так же замечательно, но просто чуть менее эффективно, так как в каждом токене придётся хранить эту пару итераторов.

Примеры использвания <code>token_def</code> и <code>token</code>:

```cpp
namespace lex = boost::spirit::lex;

// this token expose the iterator range
// boost::iterator_range of the matched input sequence
lex::token_def<> something;
// this token does not have any attribute at all
lex::token_def omit_this;
// these tokens have an associated custom attribute type
lex::token_def number;
lex::token_def identifier;

// This is the token type being used by the lexer
// as you can see, every type listed in each of the token_def's
// above, is present in the mpl type sequence as the second
// template parameter to the lex::lexertl::token type
typedef lex::lexertl::token<
    std::string::const_iterator, /*or some other base iterator type*/
    mpl::vector
> TokenT;
```

Объявленный чуть выше тип <code>TokenT</code> должен быть предоставлен соответствующему классу лексера в качестве шаблонного параметра. Из самих классов лексеров обычно используются <code>lex::lexertl::lexer</code> и <code>lex::lexertl::actor_lexer</code>, разница между ними в том, что для первого нельзя прицеплять к объявлениям токенов семантические действия (так называемые <code>actors</code>), а для второго можно, как с использованием рукописных функций, функторов, так и с помощью Boost.Phoenix, включая ленивые функции и частичное применение с помощью <code>phoenix::bind</code>.

Для токенизации используется три основных функции: <code>tokenize</code>, работающая только с лексическим анализатором, и <code>tokenize_and_parse</code> вместе с <code>tokenize_and_phrase_parse</code>, которые предназначены для использования лексера совместно с парсерами Spirit.Qi. Там всё, в принципе, достаточно похоже на то, как это выглядит в случае Spirit.Qi и функцией <code>parse</code> или <code>phrase_parse</code>, так что особо на этом останавливаться не буду. Кстати, для Spirit.Lex очень неплохо составлена <a title="Boost.Spirit.Lex documentation" href="http://www.boost.org/doc/libs/1_47_0/libs/spirit/doc/html/spirit/lex.html">документация</a>, есть много полезных и интересных примеров, которые легко приспособить к реальной жизни, всё разложено по полочкам, так что проблем с отысканием прототипов для этих функций и примеров их использования быть не должно.

Применение лексеров на практике обычно сводится к следующему:
<ol>
	<li>Пишем шаблон для класса, наследующийся от базового класса лексера (представлен классом <code>boost::spirit::lex::lexer&lt;T&gt;</code>).</li>
	<li>внутри класса делаем необходимые объявления <code>token_def</code> для используемых токенов (подобно правилам в грамматиках в Spirit.Qi).</li>
	<li>В конструкторе для каждого определения токена пишем распознаваемый паттерн с помощью регулярных выражений, записанных с помощью строк (не забываем экранировать обратные слеши — в C++ нет verbatim строк!). Справка по поводу допустимых регулярных выражений есть в <a title="supported regular expressions" href="http://www.boost.org/doc/libs/1_47_0/libs/spirit/doc/html/spirit/lex/quick_reference/lexer.html">документации</a>.</li>
	<li>Также в конструкторе добавляем определения токенов к лексеру с помощью <code>operator=</code>, <code>operator+=</code> или члена-функции <code>add</code> для поля <code>self</code>, выглядит это обычно так: <code>self.add(token1)...(tokenN);</code></li>
	<li>Далее определить тип токена для лексического анализатора и тип самого лексера, потом подсунуть получившийся базовый тип лексера в шаблонный параметр к написанному лексеру.</li>
	<li>Воспользоваться одной из функций <code>tokenize</code>, <code>tokenize_and_parse</code> или <code>tokenize_and_phrase_parse</code> для проведения лексического анализа (возможно, совместно с парсингом выражения).</li>
</ol>
Вот, в принципе и всё! За примерами можно заглянуть в официальную документацию, благо этих самых примеров там много и разобраться в них достаточно просто. Тем не менее, простенький пример лексера я всё же приведу, как некоторую заготовку для лексического анализатора, занимающегося, например, разбором какого-то простенького языка программирования:

Сначала мы определяем два типа токенов: <code>ConsumedToken</code>, которые просто съедаются при прохождении лексического анализатора, не предоставляя никакого атрибута при распознании, сюда относятся различные разделители: круглые, квадратные и фигурные скобки, знак присваивания, запятая, объявление переменной с помощью ключевого слова <code>"variable"</code>; и <code>Token</code>, которые оставляют после себя значение атрибута, сюда относятся строковые и целочисленные литералы, которые возвращают диапазон символов, задаваемый двумя итераторами, указывающими на начало и конец последовательности распознанных символов. Конечно, целочисленные литералы возможно было бы объявить и с типом атрибута вроде <code>int</code> или <code>double</code> но особой роли это не играет, необходимые преобразования типов можно сделать уже на этапе парсинга. Также есть ещё два паттерна, которые не имеют атрибута: токен, включающий в себя последовательность пробельных символов и ещё один, который включает в себя описание обычных однострочных комментариев. С ними особо ничего особенного не происходит за исключением одного момента: каждому из этих двух токенов назначено семантическое действие, которое имеет следующий вид:

```cpp
[boost::spirit::lex::_pass
    = boost::spirit::lex::pass_flags::pass_ignore];
```

В Spirit.Lex имеется набор предопределённых имён-заменителей (placeholders) для того, чтобы их можно было использовать в семантических правилах, изменяя поведение лексера по своему усмотрению. Бывают они следующие:
<ul>
	<li><code>_start</code> — итератор на начало цепочки распознанной последовательности символов.</li>
	<li><code>_end</code> — итератор, указывающий на заграничный (past the end) элемент в распознанной последовательности символов.</li>
	<li><code>_pass</code> — обозначает значение, указывающее на результат семантического действия. По-умолчанию равно значению флага <code>lex::pass_flags::pass_normal</code>. Если <code>_pass</code> установлен в значение <code>lex::pass_flags::pass_fail</code>, то лексер будет вести себя так, будто бы никакого токена не было распознано, точнее распознание токена завершилось неудачей. В случае, если флаг установлен в значение <code>lex::pass_flags::pass_ignore</code>, лексер проигнорирует текущее совпадение и продолжит пытаться дальше распознавать токены во входном потоке.</li>
	<li><code>_tokenid</code> — возвращает идентификатор для генерируемого токена.</li>
	<li><code>_val</code> — значение, из которого будет инициализирован следующий токен.</li>
	<li><code>_state</code> — состояние лексера, в котором была считана последовательность символов из входного потока.</li>
	<li><code>_eoi</code> — конечный итератор для входного потока лексера (от "end of input").</li>
</ul>
В приведённом семантическом действии фигурирует заместитель <code>_pass</code>, которому присваивается флаг <code>lex::pass_flags::pass_ignore</code>, что означает, что лексер просто будет пропускать подобные токены и сразу же пытаться распознать что-то другое. Как раз то, что нужно для комментариев! Собственно для них это семантическое действие и используется. Определение токена <code>Whitespace</code> снабжено таким же действием для того, чтобы избавиться от необходимости использовать парсер пробелов в грамматиках тогда, когда дело дойдёт до парсинга.

После описания самих паттернов через регулярные выражения мы добавляем объявления токенов к лексеру через вызов <code>self.add</code>. Далее определяем тип токена для лексера через тип <code>lex::lexertl::token</code>, и тип лексера, в качестве которого выбираем <code>lex::lexertl::actor_lexer</code> (используется именно <code>actor_lexer</code> заместо простого <code>lexer</code> потому что у нас есть парочка токенов, для которых описаны семантические действия; обычный <code>lexer</code> не имеет возможности определять семантические действия в описаниях токенов). И, в конце концов, определяем наш лексер с шаблонным параметром <code>actor_lexer&lt;TokenType&gt;</code>.

Я уже упоминал тот факт, что Spirit.Lex имеет возможности для создания статических анализаторов наряду с динамическими. Естественно, использование статического анализатора — довольно хорошая оптимизация в том плане, что нужные таблицы для разбора будут сгенерированы во время компиляции кода и нам не придётся тратить на это время в процессе исполнения программы, имея полностью готовый к работе конечный автомат. Spirit.Lex сделан таким образом, что преобразование динамического лексера в статический может быть сделано предельно простым. Для того, чтобы сгенерировать код для статического анализатора, достаточно воспользоваться функцией <code>lexertl::generate_static_dfa</code>, а потом вместо <code>lexertl::lexer</code> пользоваться классом <code>lexertl::static_lexer</code> в который одним из шаблонных параметров нужно передать сгенерированный тип статического анализатора. За наглядным примером смотрим в <a title="lexer static model" href="http://www.boost.org/doc/libs/1_47_0/libs/spirit/doc/html/spirit/lex/abstracts/lexer_static_model.html">документацию</a>, там всё чётко расписано.
<h3><strong>Чуть-чуть расширяем калькулятор.</strong></h3>
Ну и напоследок, как я обещал, маленький пример того, как можно расширять функционал калькулятора арифметических выражений, добавляя, например, свои пользовательские функции, притом достаточно простым способом. Приведённый ниже код является логическим продолжением того, чем мы занимались во <a title="часть вторая" href="http://kiri11.ru/boost_spirit_qi_part2/">второй части</a>, только с добавлением возможности использования определённого набора функций. Здесь я ограничился тем, что атрибуты выражений имеют тип <code>double</code>, но на деле можно пойти чуть дальше и зашаблонить тип для вычислений на некоторый произвольный, например, если стоит задача увеличения точности вычислений и того, что даёт тип <code>double</code> не хватает. Мне вот просто было лень это делать, так что уж как-нибудь сами, добавить дополнительный шаблонный параметр к грамматике — дело не сложное. :)
Я специально не стал добавлять все доступные функции из стандартного заголовочного файла <code>&lt;cmath&gt;</code>, ибо их там много, да и каждый желающий самостоятельно без труда может все прототипы отыскать.

Основная работа ложится на X-макросы, которые я в достаточной степени объяснял в <a title="часть третья" href="http://kiri11.ru/boost_spirit_qi_part3/">третьей части</a>. С их помощью определяются ленивые <code>phoenix</code> функции для использования в семантических действиях и сами правила с использованием этих ленивых функций. Чисто для красоты добавлена поддержка математических констант <code>pi</code> и <code>e</code> с помощью средств библиотеки <code>boost::math</code>, опять же без труда можно добавить и дополнительные константы.

Вот, собственно, и сам код:

```cpp
#define BOOST_SPIRIT_USE_PHOENIX_V3
#include 
#include 

#include 

#include 

namespace detail { namespace __impl {

#define FOREACH_UNARY_FN(apply) \\
    apply(exp)  \\
    apply(sin)  \\
    apply(cos)  \\
    apply(tan)  \\
    apply(log)  \\
    apply(sqrt) \\
    apply(abs) 

#define FOREACH_BINARY_FN(apply) \\
    apply(pow)                   

#define DEFINE_FN(id)            \\
struct id ## _impl               \\
    {                            \\
    template         \\
    struct result                \\
        {                        \\
        typedef T type;          \\
        };                       \\
                                 \\
    template         \\
    T operator() (T value) const \\
        {                        \\
        return (std::id)(value); \\
        }                        \\
    };

// define unary function_impls
FOREACH_UNARY_FN(DEFINE_FN)
#undef DEFINE_FN

#define DEFINE_FN(id)                         \\
struct id ## _impl                            \\
    {                                         \\
                                              \\
    template <                                \\
          typename This                       \\
        , typename Arg1                       \\
        , typename Arg2                       \\
    >                                         \\
    struct result           \\
        {                                     \\
        typedef Arg1 type;                    \\
        };                                    \\
                                              \\
    template      \\
    _Tx operator()    ( _Tx arg1              \\
                      , _Ty arg2) const       \\
            {                                 \\
            return (std::id)(arg1, arg2);     \\
            }                                 \\
    };

// define binary function_impls

FOREACH_BINARY_FN(DEFINE_FN)
#undef DEFINE_FN

// make unary and binary impls lazy phoenix functions
#define DEFINE_PHOENIX_LAZY_FUNCTION(id) \\
    boost::phoenix::function const id = id ## _impl();

FOREACH_UNARY_FN(DEFINE_PHOENIX_LAZY_FUNCTION)
FOREACH_BINARY_FN(DEFINE_PHOENIX_LAZY_FUNCTION)

#undef DEFINE_PHOENIX_LAZY_FUNCTION

} // namespace detail::__impl

namespace spirit = boost::spirit;
namespace qi = spirit::qi;
namespace phx = boost::phoenix;
namespace consts = boost::math::constants;

#define UNARY_RULE(id) \\
    | qi::no_case[#id] >> '(' >> expr [ qi::_val = __impl::id(qi::_1) ] >> ')'
#define BINARY_RULE(id) \\
    | (qi::no_case[#id] >> '(' >> expr >> ',' >> expr >> ')') \\
      [ qi::_val = __impl::id(qi::_1, qi::_2) ]

template 
struct calc_grammar
    : qi::grammar
    {
    calc_grammar()
        : calc_grammar::base_type(expr)
        {
        expr =
            term                [ qi::_val =  qi::_1 ]
            >> * ( '+' >> term  [ qi::_val += qi::_1 ]
                 | '-' >> term  [ qi::_val -= qi::_1 ]
                 )
            ;

        term =
            factor              [ qi::_val =  qi::_1 ]
            >> * ( '*' >> factor[ qi::_val *= qi::_1 ]
                 | '/' >> factor[ qi::_val /= qi::_1 ]
                 )
            ;

        factor =
                qi::double_     [ qi::_val =  qi::_1 ]
            |   '(' >> expr     [ qi::_val =  qi::_1 ] >> ')'
            |   '-' >> factor   [ qi::_val = -qi::_1 ]
            |   '+' >> factor   [ qi::_val =  qi::_1 ]
            |   qi::no_case["pi"] [ qi::_val = consts::pi() ]
            |   qi::no_case["e" ] [ qi::_val = consts::e()  ]
                FOREACH_UNARY_FN(UNARY_RULE)
                FOREACH_BINARY_FN(BINARY_RULE)
            ;
        }

    qi::rule
        expr, term, factor;
    };

#undef UNARY_RULE
#undef BINARY_RULE

#undef FOREACH_UNARY_FN
#undef FOREACH_BINARY_FN
} // namespace detail
```
