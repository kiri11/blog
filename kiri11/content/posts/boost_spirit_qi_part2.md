<h2><strong>Учимся извлекать из примеров пользу. Новые друзья — атрибуты.</strong></h2>
Вот и вторая часть повествования, здесь мне придётся вас маленько огорчить — сразу же переходить ко второму пункту нашего плана я не стану, а прежде расскажу чуть более подробно о синтезируемых атрибутах, это нам очень пригодится в будущем. Понимание всех этих основных принципов для освоения Спирита очень важно, так что я просто не могу обойти эту тему стороной и сразу поместить читателя в гущу событий, чтобы он "как-нибудь там" разбирался с тем, что так внезапно свалилось на голову.<!--more-->
<h3><strong>Сигнатура правил и грамматик.</strong></h3>
Как я уже говорил, сигнатуры выглядят следующим образом: <code>RT(A1, ..., AN)</code>, где: <code>RT</code> — тот атрибут, который в результате возвращает само правило; <code>A1, ..., AN</code> — атрибуты-параметры, то есть те величины, от которых зависит правая часть правила (обычно их называют "наследуемые атрибуты", от англ.: inherited attributes).

Надеюсь, простят мне читатели некоторую вольность, когда я говорю о правилах и грамматиках одновременно, тем временем, упоминая лишь что-то одно: грамматика, грубо говоря, является лишь более высокоуровневым "подобием" правила, семантика у них весьма и весьма схожая, так что я не вижу особого смысла оговаривать каждый случай в отдельности. К слову, в Спирите есть и ещё одна грамматического плана сущность — подправила (<code>subrules</code>), более узкие, нежели обыкновенные правила, но их рассмотрение выходит за рамки данной темы, просто чтобы вы знали, если кому уж очень интересно станет поглядеть самому.
<h3><strong>Синтезируемые атрибуты.</strong></h3>
Итак, синтезируемые атрибуты — это то, что возвращает нам успешно отработавший парсер. Например, если парсер <code>double_</code> распознает число с плавающей точкой <code>1.5</code>, то он вернёт в качестве такого атрибута то самое число <code>1.5</code> типа <code>double</code>, что было распознано. В наших правилах мы можем контролировать как атрибуты каждого из парсеров, входящих в правую часть правила, так и синтезируемый атрибут левой части — самого правила. Для манипуляций с этими атрибутами было создано несколько имён-заместителей (англ.: <code>placeholders</code>; то есть таких переменных, подставив которые в нужное место и в нужное время, <code>spirit</code> сам разберется, что вместо заместителя передавать):
<ul>
	<li><code>qi::_val</code> — синтезируемый атрибут левой части правила.</li>
	<li><code>qi::_1</code> — атрибут, содержащий результат работы парсера.</li>
	<li><code>qi::_2, qi::_3, ...</code> — дополнительные имена-заместители, предназначенные для различных нужд (например, обработка ошибок).</li>
</ul>
В основном, с этими placeholder'ами мы имеем дело в семантических действиях, так что сейчас познакомимся и с ними.
<h3><strong>Семантические действия.</strong></h3>
Каждому парсеру Спирита возможно назначить семантическое действие, которое будет выполнено сразу же, как только успешно отработает парсер, к которому оно присоединено. Запись такая: <code>P[F]</code>, где: <code>P</code> — парсер, <code>F</code> — семантическое действие, присоединённое к парсеру <code>P</code>.

На деле, <code>F</code> может быть как обычной функцией, так и функтором. В первом случае сигнатура функции может выглядеть так:

```cpp
void F(Attrib const&);
void F(Attrib const&, Context&);
void F(Attrib const&, Context&, bool&);
```

Последние два параметра нам с вами будут не нужны, так что мы просто не будем их использовать, остановившись на самом первом варианте

```cpp
void F(Attrib const&);
```

где принимается ссылка на константный атрибут, полученный от сработавшего парсера.

Для функторов сигнатура следующая:

```cpp
void operator()(Attrib const&, unused_type, unused_type) const;
void operator()(Attrib const&, Context&, unused_type) const;
void operator()(Attrib const&, Context&, bool&) const;
```

Точно так же, последние два параметра мы можем проигнорировать, беря во внимание лишь тот атрибут, что попадает к нам в руки для дальнейших манипуляций.

Особенно хорошо выглядят действия с использованием <code>boost::phoenix</code>, но, дабы не создавать лишних сложностей и не валить всё в одну кучу, пока избежим разговора о нём.
<h3>Небольшой примерчик для начала:</h3>
Например, требуется распознавать строки такого вида: <code>[integer]</code>, где <code>integer</code> — целое число.

Мы можем написать примерно следующее для решения этой задачки:

```cpp
qi::parse(begin, end, '[' >> qi::int_ >> ']');
```

Отлично, но нам бы ещё, например, вывести это число в лог какой-нибудь, а не просто считать и проигнорировать результат. Здесь и пригодятся семантические действия. У нас пока что есть пара вариантов: написать простую функцию для вывода числа в лог, либо же сделать функтор для решения данной задачи. Сделаем и так, и так, дабы показать различия в синтаксисе в том и в другом случае:

```cpp
// plain function
void to_log(int const & val)
    {
    std::clog << val << std::endl;
    }

// function object
struct to_log_functor
    {
    void operator() ( int const & val
                    , qi::unused_type
                    , qi::unused_type ) const
        {
        std::clog << val << std::endl;
        }
    };
```

В случае описания функтора, <code>operator()</code> должен иметь ровно три параметра, от этого никуда не деться, хотя позже мы узнаем, что есть и немного другой способ, который этого не требует, с использованием всё того же <code>boost::phoenix</code>.

Теперь наконец посмотрим, как прицепить выполнение действия при парсинге в первом случае (если у нас есть простая функция, то передавать надо указатель на неё):

```cpp
qi::parse(begin, end, '[' >> qi::int_[&to_log] >> ']');
```

и во втором случае, с функтором:

```cpp
qi::parse(begin, end, '[' >> qi::int_[to_log_functor()] >> ']');
```

Стоит также упомянуть тот факт, что возможно использовать в качестве действий и функции-члены классов, но тут опять всё тот же феникс возникает, так что нам пока хватит и того, что уже есть.

Кстати, как вы уже, должно быть, заметили, я описывал правило для парсинга прямо в функции parse, что подтверждает то, что абсолютно все сущности <code>spirit::qi</code>, принимающие участие в парсинге, сами трактуются как парсеры и могут быть использованы в любом месте, где ожидается парсер <code>spirit::qi</code>.

Вот теперь мы получаем то, что нужно, вывод в лог осуществлён. Впрочем, хотелось бы ещё и число это самое получить в качестве результата, чтобы потом что-то с ним делать. Но вот здесь нас могут поджидать небольшие неприятности. Несмотря на то, что прямая попытка получить из предыдущего кода результат парсинга оказывается успешной:

```cpp
std::string str = "[1]";
std::string::iterator begin = str.begin(), end = str.end();
int result;
qi::parse(begin, end, '[' >> qi::int_[to_log_functor()] >> ']', result);
// here result = 1;
```

Если мы попытаемся рефакторить наш код в нормальную грамматику, получим совсем другие результаты:

```cpp
template <typename Iterator>
struct simple_grammar : qi::grammar<Iterator, int()>
    {
    simple_grammar()
        : simple_grammar::base_type(start)
        {
        start = '[' >> qi::int_[to_log_functor()] >> ']';
        }

    qi::rule<Iterator, int()> start;
    };

...

std::string str = "[1]";
std::string::iterator begin = str.begin(), end = str.end();

simple_grammar<std::string::iterator> gr;
int result;

qi::parse(begin, end, gr, result);
// here result is undefined!
```

Почему такое произошло, что случилось?! Мы ведь явно написали, что у грамматики есть атрибут целочисленного типа, обеспечили функции <code>parse</code> необходимое число для записи атрибута парсинга туда! А штука в том, что если в правой части правила есть хотя бы одно семантическое действие, то автоматическое присваивание атрибута правой части атрибуту левой игнорируется. В принципе, можно поступить двумя способами: либо написать семантическое действие, которое будет присваивать нужный атрибут левой части (запишется это так: <code>[qi::_val = qi::_1]</code>), либо воспользоваться очень удобной фичей Спирита под названием "автоматические правила" (<code>auto rules</code>). В случае с прямой подстановкой выражения в функцию parse дело сработало именно потому, что там действует автоматическое распространение атрибутов (<code>attribute propagation</code>) по-умолчанию.
<h3><strong>Автоматические правила.</strong></h3>
Довольно часто можно видеть правила вида <code>R = P[qi::_val = qi::_1]</code>, так вот если у нас как раз такое правило и атрибут правой части правила совместим с атрибутом левой части, то можно воспользоваться такой записью: <code>R %= P</code>, так что автоматически атрибут парсера <code>P</code> присваивается атрибуту правила <code>R</code>. Удобно! В нашем случае код начального нетерминала грамматики можно переписать так:

```cpp
start %= '[' >> qi::int_[to_log_functor()] >> ']';
```

Что будет эквивалентно такому коду:

```cpp
start = ('[' >> qi::int_[to_log_functor()] >> ']')[qi::_val = qi::_1];
```

Теперь всё работает и мы можем каким-либо образом использовать полученные данные, дело сделано. Впрочем, одно-то число распознать не проблема. Как быть, если нам нужно распознать пару чисел, целый список, куда и как их сохранять? Прежде, чем ответить на этот вопрос, ещё попутно освещу один момент, связанный с тем, какие атрибуты могут иметь сложные выражения в <code>spirit</code>.
<h3><strong>Атрибуты сложных выражений.</strong></h3>
До сих пор мы рассматривали только одиночные атрибуты, каждый сам по себе. А что будет атрибутом выражения <code>"A &gt;&gt; B", "A|B", "*A"</code>, например? Вообще говоря, зависит от того, какими атрибутами будут обладать парсеры <code>A</code> и <code>B</code>. Не стану молотить отсебятину, а просто буду приводить отрывочки из официальной документации, которая довольно наглядно показывает, во что превращаются атрибуты парсеров в сложных выражениях. Например:

```cpp
Sequence attribute propagation rule
a: A, b: B --> (a >> b): tuple<A, B>
```

Теперь что это означает: если парсер <code>a</code> обладает атрибутом типа <code>A</code>, а парсер <code>b</code> — атрибутом типа <code>B</code>, то результат <code>(a &gt;&gt; b)</code> будет иметь атрибут типа <code>tuple&lt;A, B&gt;</code>. Что за <code>tuple</code>, откуда он взялся? В данном случае, запись <code>tuple&lt;A, B&gt;</code> является просто заместителем любого контейнера библиотеки <code>boost::fusion</code>, который может держать в себе типы <code>A</code> и <code>B</code>, как например <code>boost::fusion::tuple&lt;A, B&gt;</code>, <code>boost::fusion::vector&lt;A, B&gt;</code>, <code>std::pair&lt;A, B&gt;</code>. С <code>fusion</code> особо знакомиться пока не будем, просто знайте, что есть такая библиотека, контейнеры которой используются здесь, чуть позже объясню, почему используются именно они.

В принципе, очевидно, что для того, чтобы предоставляемый тип был совместим с типом атрибута, он должен:
<ul>
	<li>либо быть конвертируемым в тип атрибута (например, <code>int</code> конвертируется в <code>double</code>),</li>
	<li>либо представлять определённую функциональность, в частности, следовать концепции, совместимой со сложным типом атрибута.</li>
</ul>
Далее будет также использоваться несколько других обозначений: <code>vector&lt;A&gt;</code> — заменитель для любого STL совместимого контейнера, способного держать элементы типа <code>A</code>, <code>Unused</code> заменяет тип <code>unused_type</code>.

```cpp
Sequence (a >> b):

a: A, b: B --> (a >> b): tuple<A, B>
a: A, b: Unused --> (a >> b): A
a: Unused, b: B --> (a >> b): B
a: Unused, b: Unused --> (a >> b): Unused

a: A, b: A --> (a >> b): vector<A>
a: vector<A>, b: A --> (a >> b): vector<A>
a: A, b: vector<A> --> (a >> b): vector<A>
a: vector<A>, b: vector<A> --> (a >> b): vector<A>
```


```cpp
Alternative (a | b):

a: A, b: B --> (a | b): variant<A, B>
a: A, b: Unused --> (a | b): optional<A>
a: A, b: B, c: Unused --> (a | b | c): optional<variant<A, B> >
a: Unused, b: B --> (a | b): optional<B>
a: Unused, b: Unused --> (a | b): Unused
a: A, b: A --> (a | b): A
```


```cpp
Kleene (*a):

a: A --> *a: vector<A>
a: Unused --> *a: Unused
```

Так, ещё какие-то <code>variant</code> и <code>optional</code> появились, это что такое... Не думаю, что должны бы возникнуть такие вопросы, но, тем не менее, расскажу в общих чертах, что за этими названиями скрывается. <code>Variant&lt;A, B&gt;</code> — заменитель для контейнера, способного держать в один момент времени лишь одну сущность, принадлежащую одному из типов, перечисленных в шаблонных параметрах, реальный тип такого контейнера будет <code>"boost::variant&lt;A, B&gt;"</code>, эдакий <code>union</code>, только типобезопасный и с кучей разных вкусностей. <code>Optional&lt;A&gt;</code> — заместитель для контейнера, который либо может содержать в себе значение типа <code>A</code>, либо может вообще ничего не содержать(англ.: <code>optional</code> — необязательный), например <code>boost::optional&lt;A&gt;</code>. <code>Unused</code> просто означает то, что тип атрибута данного выражения не используется.

Разберёмся поподробней с тем, что тут происходит. Начнём с оператора следования: если один из атрибутов оператора не используется, то результатом следования будет второй атрибут, если оба типа не используются, то и результат тоже не имеет атрибута, довольно естественное упрощение. Если типы обоих атрибутов представляют собой коллекции из элементов одинаковых типов, то в результате атрибутом станет тоже коллекция элементов того же самого типа. Что это означает на практике: если у <code>a</code> есть атрибут <code>std::vector&lt;int&gt;{1, 2, 3}</code>, а у <code>b</code> — <code>std::vector&lt;int&gt;{4, 5, 6}</code>, то результат <code>a &gt;&gt; b</code> будет вектором <code>std::vector&lt;int&gt;{1, 2, 3, 4, 5, 6}</code>, они просто "склеятся" вместе. А вот когда типы <code>A</code> и <code>B</code> разные и не конвертируются друг в друга неявно, придётся просто "свалить всё это дело в кучу" и использовать контейнер для хранения гетерогенных коллекций данных типа того же <code>tuple&lt;A, B&gt;</code>, как раз поэтому используются контейнеры <code>fusion</code>, которые позволяют содержать элементы разных типов вместе.

После объяснений по поводу <code>variant</code> и <code>optional</code> думаю, что разбор вариантов с оператором альтернативы не вызовет трудностей: все правила абсолютно естественны, единственное, что следует отметить: если <code>A</code> и <code>B</code> — одинаковые типы, то результатом <code>a | b</code> станет просто <code>A</code>, а не какой-нибудь <code>variant&lt;A, A&gt;</code>.

С оператором звёздочки всё просто: контейнер <code>vector&lt;A&gt;</code> последовательно набивается теми элементами, что распознаёт парсер <code>a</code>.

А вот и ещё одна мелочь нарисовалась: кто-то наверняка уже заметил, что, говоря про <code>tuple&lt;A, B&gt;</code>, в качестве реальных примеров контейнеров <code>fusion</code> я упоминал <code>std::pair&lt;A, B&gt;</code> — вроде не похоже на контейнер <code>fusion</code>, ага? Всем нам знакомая пара из стандартной библиотеки. Так-то оно, конечно, так, но этот тип используется довольно часто, в следствие чего для пары решено было сделать совместимые с <code>boost::fusion</code> обёртки "прямо из коробки". Самое время рассмотреть связанные с этим примеры.
<h3><strong>Разбор пары значений.</strong></h3>
Рассмотрим небольшой пример с парсингом пары чисел вида <code>[integer, integer]</code> в структуру <code>std::pair&lt;int, int&gt;</code>.

Естественно, первое, что мы сделаем с нашим предыдущим примером — захотим переписать всё это дело примерно так:

```cpp
template <typename Iterator>
struct simple_grammar
    : qi::grammar<Iterator, std::pair<int, int>()>
    {
    simple_grammar()
        : simple_grammar::base_type(start)
        {
        start = '[' >> qi::int_ >> ',' >> qi::int_ >> ']';
        }

    qi::rule<Iterator, std::pair<int, int>()> start;
    };
```

Теперь посмотрим, что творится: атрибутом правой части правила <code>start</code> будет некий контейнер типа <code>vector&lt;int&gt;</code>, вроде не <code>pair</code>, да?

Если мы попробуем скомпилировать следующий код с нашей грамматикой

```cpp
std::string str = "[1,2]";
std::string::iterator begin = str.begin(), end = str.end();
simple_grammar<std::string::iterator> gr;
std::pair<int, int> p;
qi::parse(begin, end, gr, p);
```

то ничего не получится, будут ошибки в стиле <code>'невозможно преобразовать тип ... в std::pair&lt;int, int&gt;'</code>. Всё из-за того, что <code>std::pair&lt;T1, T2&gt;</code> не является стандартным контейнером <code>fusion</code> и тот тип, который получился в результате операций <code>qi::int_ &gt;&gt; ',' &gt;&gt; qi::int_</code> невозможно прозрачно преобразовать в <code>std::pair&lt;int, int&gt;</code>. Впрочем, я же что-то говорил про то, что для <code>std::pair&lt;T1, T2&gt;</code> есть обёртка уже в составе самой библиотеки <code>fusion</code>. Какая удача, воспользуемся этим! Всё, что нужно для того, чтобы сделать <code>std::pair&lt;A, B&gt;</code> полноценным членом <code>boost::fusion</code>, надо подключить заголовок

```cpp
<boost/fusion/include/std_pair.hpp>
```

Теперь приведённый пример будет компилироваться успешно, так как мы предоставили всё необходимое для того, чтобы <code>fusion</code> смог без посторонней помощи переделать свой тип в <code>std::pair&lt;T1, T2&gt;</code>.

К слову, возможен и такой вариант: функции <code>parse</code> и <code>phrase_parse</code>, как мы видели в предыдущей части из их прототипов, могут принимать в качестве выходных атрибутов <code>N</code> величин (на самом деле, это <code>N</code> устанавливается равным значению препроцессорной константы <code>SPIRIT_ARGUMENTS_LIMIT</code>, которая по-умолчанию равна <code>10</code>), так вот без необходимости подключать предыдущий заголовок, следующий код отработает вполне успешно:

```cpp
std::string str = "[1,2]";
std::string::iterator begin = str.begin(), end = str.end();
std::pair<int, int> p;
qi::parse(begin, end, '['
                      >> qi::int_ >> ',' >> qi::int_ >>
                      ']', p.first, p.second);
```

Таким образом, имеем возможность работать с функциями <code>parse</code> и <code>phrase_parse</code> в <code>printf</code>-подобной манере, иногда это может оказаться очень удобным.

Работа с автоматическим заполнением атрибутов сводится обычно к тому, чтобы адаптировать нашу пользовательскую структуру для того, чтобы она могла считаться полноправным <code>fusion</code> контейнером, ибо большинство производимых спиритом коллекций атрибутов на деле являются <code>fusion</code> последовательностями, а в обычные контейнеры типа <code>std::pair&lt;T1, T2&gt;</code> они уже могут прозрачно переделываться благодаря тому, что это предусмотрено в самой библиотеке <code>fusion</code>.

Теперь мы умеем заполнять пары, основываясь на знаниях об атрибутах. Уже что-то, впрочем, иногда хочется оперировать с последовательностями произвольной длины, то есть иметь на выходе, например, вектор, состоящий из некоторых данных.
<h3><strong>Парсинг списка величин.</strong></h3>
Рассмотрим такой пример: дана строка вида <code>{&lt;id1&gt;:&lt;num1&gt;, &lt;id2&gt;:&lt;num2&gt;, &lt;id3&gt;:&lt;num3&gt;, ..., &lt;idN&gt;:&lt;numN&gt;}</code> и нам нужно вытащить из неё все входящие в этот набор числа.

Как поступим: можно было бы просто считывать все символы в строке до очередного <code>':'</code>, забирать число, а потом просто составить из этого дела список величин, разделённых запятыми. Но, вероятно, в перспективе мы бы хотели с минимальными переделками кода иметь возможность наряду с числами вытаскивать и соответствующие идентификаторы, так что можно поступить более интересным способом, распознавая валидные идентификаторы впридачу.

Более простой вариант мог бы выглядеть так:

```cpp
bool ParseNumbers( std::string & str
                 , std::vector<int> & nums)
    {
    std::string::iterator  begin = str.begin()
                         , end   = str.end();

    namespace qi = boost::spirit::qi;
    return qi::parse(begin, end,
                (
                '{' >>
                    (
                    qi::omit[*(qi::char_ - ':')] >> ':'
                    >> qi::int_
                    ) % ','
                >> '}'
                ), nums
            ) && begin == end;
    }
```

Но он плохой в том плане, если вместо нормального идентификатора у нас будет какой-то мусор вплоть до каждого двоеточия — нехорошо. А тем не менее, мы здесь видим нечто новое, об этом стоит рассказать. Первое, что можно заметить — парочка новых операторов. Сначала видим выражение <code>qi::char_ - ':'</code> — это оператор "разности", то есть в данном случае получившийся парсер будет считывать все символы (<code>qi::char_</code>), кроме двоеточия, таким образом, этот оператор позволяет регулировать поведение парсера, говоря ему: "делай то, что должен, только кроме вот этого", иначе, позволяет задать список исключения определённых результатов для некоторого парсера. Дальше видим, что всё это дело обёрнуто в директиву <code>qi::omit[]</code>, это директива, которая заставляет игнорировать возвращаемый атрибут парсера, который внутри находится, иными словами, вместо возвращаемого атрибута от парсера мы получим <code>unused_type</code>. То есть: <code>*(qi::char_ - ':')</code> даст атрибут типа <code>vector&lt;char&gt;</code>, но нам эта информация пока ни к чему и мы просто хотим проигнорировать эту поступившую информацию точно так же, как например, игнорируем возвращаемое значение от фигурных скобок для этого выражения или двоеточия — они нам не важны. Итого, что имеем пока:

```cpp
qi::omit[*(qi::char_ - ':')] >> ':'
>> qi::int_
--> int
```

Здесь считываем все символы до двоеточия, само двоеточие съедаем, а потом забираем число, атрибутом этого всего выражения будет значение типа <code>int</code>, то есть из всей этой информации мы получаем в качестве результата лишь число.

Дальше нам встречается ещё один новый оператор — оператор списка <code>'%'</code>. Что он делает: пусть дано выражение <code>'A % B'</code>, тогда данный парсер распарсивает список из величин, распознаваемых парсером <code>A</code>, разделённых величинами, распознаваемыми парсером <code>B</code>. Иначе говоря, его действие эквивалентно следующему выражению:

```cpp
A >> *(B >> A)
```

Что у нас есть в нашем примере: <code>&lt;идентификатор с двоеточием и число&gt; % ','</code>. Расшифровываем: список из <code>&lt;идентификатор с двоеточием и число&gt;</code>, разделённых между собой запятыми. К слову, атрибутом парсера <code>'a % b'</code> является <code>vector&lt;A&gt;</code>, где <code>A</code> — тип атрибута для парсера <code>а</code>. В нашем случае, атрибутом этого сложного выражения является атрибут <code>vector&lt;int&gt;</code>, потому что <code>&lt;идентфикатор с двоеточием и число&gt;</code> имеет атрибут типа <code>int</code>, благодаря директиве <code>qi::omit[]</code>. Таким образом, атрибут всего выражения таков:

```cpp
'{' >>
	(
	qi::omit[*(qi::char_ - ':')] >> ':'
	>> qi::int_
	) % ','
>> '}'
--> vector<int>
```

В функцию <code>parse</code> мы передаём <code>std::vector&lt;int&gt;</code>, а тип атрибута у нас является STL совместимым контейнером с элементами типа <code>int</code>, так что возможно прозрачное присвоение атрибута нашему результирующему вектору.

Вот, собственно, познакомились с тем, как заполнять пары чисел и векторы, узнали о новых операторах списка (<code>%</code>) и разности (<code>-</code>), увидели новую директиву <code>qi::omit[]</code>, позволяющую игнорировать распространение атрибута для парсера.

А теперь посмотрим на более идеологически правильный вариант:

```cpp
bool ParseNumbers( std::string & str
                 , std::vector<int> & nums)
    {
    std::string::iterator  begin = str.begin()
                         , end   = str.end();

    namespace qi = boost::spirit::qi;
    return qi::phrase_parse(begin, end,
                (
                '{' >>
                    (
                    qi::omit[
                        qi::lexeme[
                            (qi::alpha | '_')
                            >> *(qi::alnum | '_')
                        ]
                    ] >> ':'
                    >> qi::int_
                    ) % ','
                >> '}'
                ), qi::space, nums
            ) && begin == end;
    }
```

Почти то же самое, за исключением парочки моментов. Первое — мы использовали функцию <code>phrase_parse</code> вместо <code>parse</code> и добавили в качестве стандартного парсера пропусков <code>qi::space</code>, пропускающий самые обыкновенные пробельные символы, включая символы новой строки и табуляции. То есть теперь мы не обязаны писать всю строку исключительно без пробелов, чтобы парсинг проходил успешно, теперь можно так:

```cpp
{id1:124,id2:222}
```

или так:

```cpp
{  id1   :
123 , id2 : 4 }
```

Второе, что было изменено: считывание идентификатора теперь происходит иначе:

```cpp
(qi::alpha | '_') >> *(qi::alnum | '_')
```

<code>qi::alpha</code> распознаёт только буквенные символы, <code>qi::alnum</code> распознаёт буквы и цифры, здесь всё аналогично нотации из регулярных выражений. Также мы должны оставить возможность начинать идентификатор с знака нижнего подчёркивания и использовать его в записи строки далее, короче, строим обычный идентификатор в C-стиле. Но вместе с этим есть и ещё одно изменение: появилась наша старая знакомая — директива <code>lexeme[]</code>:

```cpp
qi::lexeme[(qi::alpha | '_') >> *(qi::alnum | '_')]
```

Особо внимательные читатели наверняка помнят, что директива <code>lexeme[]</code> отключает использование парсера пробелов для того парсера, что в эту директиву заключён. Если бы мы оставили считывание идентификатора без этой директивы, то вполне валидными оказались бы идентификаторы вида <code>'a 1', '_fucking shit'</code>, а нам такого не надо, вот мы и запрещаем неявный пропуск пробелов, дабы идентификаторы не содержали в себе пропусков.
<h3><strong>Разбор списка пар "ключ=значение".</strong></h3>
И напоследок маленький пример на случай того, если вам потребуется создавать какие-нибудь простые конфигурационные файлы, где настройки будут храниться в виде строк формата <code>"ключ=значение"</code>, а возиться с написанием велосипедов жутко не хочется или других инструментов, выполняющих нужную задачу вы попросту не знаете.

Предположим, что конфигурационный файл имеет следующий формат:

```cpp
// так обозначаются комментарии -- стандартный C++ стиль
option1;
option2 = value2;
// ...
option_N = value_N;
```

То бишь каждая пара отделяется от другой точкой с запятой, есть возможность предоставления однострочных комментариев, ключи и значения представляют собой обыкновенные числобуквенные строки. Плюс к тому, необязательно, чтобы ключ имел значение.

Недолго думая, пишем примерно следующий код:

```cpp
template <typename Iterator>
struct key_value_grammar
    : qi::grammar<
          Iterator
        , std::map<std::string, std::string>()
        >
    {
    key_value_grammar()
        : key_value_grammar::base_type(pairs)
        {
        pairs = pair % ';' >> ';';
        pair  = key >> -('=' >> value);
        key   = (qi::alpha | qi::char_('_')) >> *(qi::alnum | qi::char_('_'));
        value = +(qi::alnum | qi::char_('_'));
        }

    qi::rule<
          Iterator
        , std::map<std::string, std::string>()
    > pairs;

    qi::rule<
          Iterator
        , std::pair<std::string, std::string>()
    > pair;

    qi::rule<Iterator, std::string()> value, key;
    };
```

Здесь появилось немного нового. Во-первых, нетерминалы <code>key</code> и <code>value</code> имеют тип <code>std::string</code>, что сразу же даёт понять, что атрибут от правых частей (который представляется в виде некоторого типа <code>vector&lt;char&gt;</code>) может прозрачно преобразовываться в <code>std::string</code> — удобно! Точно такая же фигня и с нетерминалом <code>pairs</code> — его атрибут есть некий тип вроде <code>vector&lt;pair&lt;std::string, std::string&gt; &gt;</code>, так вот этот вектор тоже незаметно может конвертироваться в контейнер <code>std::map&lt;std::string, std::string&gt;</code>. Ещё увидели оператор унарного минуса, о котором я говорил ещё в предыдущей части, обозначающий то же самое, что и оператор <code>'?'</code> в стандартной записи РБНФ, то есть "может встречаться 0 или 1 раз" — это нам обеспечивает возможность не предоставлять ключу значение, если оно не нужно.

Вот ещё что: для распознания чисел, букв и нижнего подчёркивания я пишу следующий код:

```cpp
*(qi::alnum | qi::char_('_'))
```

В принципе использование оператора альтернативы здесь может быть исключено, если воспользоваться тем, что в качестве параметра парсеру <code>qi::char_</code> можно указать строку допустимых для распознавания символов в формате, соответствующем в некотором роде регулярным выражениям. Эквивалентный код с использованием этой фичи:

```cpp
*qi::char_("a-zA-Z0-9_")
```

Раз уж речь пошла о символах, то расскажу и ещё об одной полезной фиче. До настоящего момента, когда мы хотели исключить из списка распознаваемых символов какой-то определённый символ, мы пользовались бинарным оператором разности, например, <code>qi::char_ - ':'</code>. Оказывается, сделать исключение некоторых недопустимых символов можно и иначе: с помощью унарного оператора отрицания '~'. Применим этот оператор только к символьным парсерам, то есть к таким, которые считывают лишь один символ, как, например, <code>qi::char_</code>, <code>qi::digit</code>, <code>qi::alpha</code> и тому подобные. Так вот, парочка примеров действия данного оператора:

```cpp
~qi::char_ // не принимает вообще никакие символы
~qi::digit // принимает любые символы, кроме цифр
~qi::char_("a-zA-Z")
// принимает всё за исключением символов,
// входящих в диапазоны a-z и A-Z,
// короче, всё, кроме маленьких и больших английских букв
```

<h3><strong>Добавляем комментарии.</strong></h3>
Вроде бы всё есть, да только вот про комментарии совсем забыли! Нам нужно пропускать как обычные пробелы, символы новой строки и табуляции, так и однострочные комментарии, начинающиеся с последовательности "//" и заканчивающиеся концом строки. Можно было бы, конечно, написать велосипед, написав правило для комментариев my_comment и вставляя строчки вида <code>"*(qi::space | my_comment)"</code> перед и после каждого парсера в нашей грамматике, но это ни разу не удобно, да и вообще фигня какая-то. Пойдём другим путём — напишем грамматику пропусков, которая будет учитывать пробельные символы и комментарии и передадим её в качестве шаблона нашей грамматике.

Сначала напишу необходимый код, а далее дам необходимые разъяснения:

```cpp
namespace qi = boost::spirit::qi;
namespace repository = boost::spirit::repository;

template <typename Iterator>
struct skip_grammar : public qi::grammar<Iterator>
    {
    skip_grammar()
        : skip_grammar::base_type(skip)
        {
        skip
            =   qi::space
            |   repository::confix("//", qi::eol)
                [*(qi::char_ - qi::eol)]
            ;
        }

    qi::rule<Iterator> skip;
    };
```

Теперь посмотрим, что происходит: в правиле <code>skip</code> мы считываем либо обычные пробельные символы с помощью стандартного парсера <code>qi::space</code>, который этим занимается, либо парсим то, что распознаёт эта интересная конструкция:

```cpp
repository::confix("//", qi::eol)
[*(qi::char_ - qi::eol)]
```

Перед нами директива <code>confix(prefix, suffix)[subject]</code>, её действие эквивалентно следующему коду:

```cpp
qi::omit[prefix] >> subject >> qi::omit[suffix]
```

То есть данная директива даёт возможность заключить парсер <code>subject</code> между двумя парсерами <code>prefix</code> и <code>suffix</code>, при этом их атрибуты отклоняются, а в качестве атрибута выражения возвращается атрибут заключённого посередине парсера <code>subject</code>. Теперь разбираем эту директиву на нашем примере. <code>qi::eol</code> — парсер, распознающий символ конца строки. <code>confix("//", qi::eol)</code> — заключаем парсер между последовательностью символов "//" и концом строки, какой парсер у нас внутри — <code>*(qi::char_ - qi::eol)</code> — то есть абсолютно любые последовательности символов, исключая лишь символ конца строки, как раз то, что нужно для комментария.

В принципе, можно было бы написать и в более развёрнутом варианте с использованием директивы <code>qi::omit[]</code>, но, во-первых, раз есть компонент, идеально подходящий под наши цели, то стоит им воспользоваться, а не строить велосипед, а во-вторых, использование <code>confix[]</code> позволяет инкапсулировать парсеры <code>prefix</code> и <code>suffix</code> в отдельную конструкцию, скрывающую ненужные детали, что в самописном варианте не получится сделать столь красиво и экономично, как в случае с <code>confix[]</code>.

Кстати, совершенно никакого труда не стоило бы добавить и комментарии в стиле C:

```cpp
repository::confix("/*", "*/")[*(qi::char_ - "*/")]
```

Ну и маленькая мелочь: директива <code>confix[]</code> находится в пространстве имён <code>boost::spirit::repository</code> и для того, чтобы она была доступна, нужно подключить заголовок

```cpp
<boost/spirit/repository/include/qi_confix.hpp>
```

Теперь у нас есть грамматика для пропусков, дело за малым: нужно добавить использование этого парсера для нужных правил и для самой грамматики, а потом передать объект нашей грамматики <code>skip_grammar</code> в функцию <code>phrase_parse</code> вместе с целевой грамматикой.

Модифицируем <code>key_value_grammar</code>:

```cpp
template <typename Iterator>
struct key_value_grammar
    : qi::grammar<
          Iterator
        , std::map<std::string, std::string>()
        , skip_grammar<Iterator>
        >
    {
    key_value_grammar()
        : key_value_grammar::base_type(pairs)
        {
        pairs = pair % ';' >> ';';
        pair  = key >> -('=' >> value);
        key   = (qi::alpha | qi::char_('_')) >> *(qi::alnum | qi::char_('_'));
        value = +(qi::alnum | qi::char_('_'));
        }

    qi::rule<
          Iterator
        , std::map<std::string, std::string>()
        , skip_grammar<Iterator>
    > pairs;

    qi::rule<
          Iterator
        , std::pair<std::string, std::string>()
        , skip_grammar<Iterator>
    > pair;

    qi::rule<Iterator, std::string()> value, key;
    };
```

Управляющий код:

```cpp
std::string config;
// fill in 'config' string...
std::string::iterator  begin = config.begin()
                     , end   = config.end();
key_value_grammar<std::string::iterator> gr;
skip_grammar<std::string::iterator> sk;
std::map<std::string, std::string> conf_map;
qi::phrase_parse(begin, end, gr, sk, conf_map);
// ...
```

Таким образом, для того, чтобы поддерживать парсеры для символов, которые должны считаться пробельными и их атрибуты нам должны быть не важны, нужно:
<ol>
	<li>Написать грамматику, распознающую язык для всего того, что может называться комментарием и атрибуты чего мы должны игнорировать.</li>
	<li>Добавить тип пробельной грамматики в заголовки тех правил и грамматик, где данный парсер-пропускатель должен использоваться по-умолчанию.</li>
	<li>Передать объект грамматики пробелов вместе с нужной грамматикой в контролирующую функцию <code>phrase_parse</code>.</li>
</ol>
И напоследок пару слов на тему того, как можно в выбранном куске правила локально включить или отключить использование парсера пробелов.

Отключение автоматического парсера пробелов возможно сделать двумя способами: либо директивой <code>lexeme[]</code>, о которой мы уже знаем и которой успели воспользоваться, либо директивой <code>no_skip[]</code>. Разница между ними заключается лишь в одном: директива <code>lexeme[]</code> всегда, прежде чем запретить пропуск пробелов, делает предварительный пропуск символов в начале строки, то есть, например, следующий код отработает успешно и мы получим нормально распознанную строку:

```cpp
std::string expression = "   12345";
std::string::iterator  begin = expression.begin()
                     , end   = expression.end();
std::string integer;
bool success
        = qi::phrase_parse( begin
                          , end
                          , qi::lexeme[
                                -(qi::lit('+') | '-')
                                >> +qi::digit
                            ]
                          , qi::space
                          , integer
                          );
// here: success = true; integer = "12345";
```

Директива <code>no_skip[]</code> в отличие от <code>lexeme[]</code> никогда не делает предварительный пропуск символов, прежде чем отключить парсер пробелов, во всём остальном она абсолютно идентична директиве <code>lexeme[]</code>. Например, тот же самый код, но с использованием директивы <code>no_skip[]</code> заместо <code>lexeme[]</code> завершится неудачей, потому что пробелы в начале строки не будут автоматически считаны как это было бы сделано в случае с <code>lexeme[]</code>:

```cpp
std::string expression = "   12345";
std::string::iterator  begin = expression.begin()
                     , end   = expression.end();
std::string integer;
bool success
        = qi::phrase_parse( begin
                          , end
                          , qi::no_skip[
                                -(qi::lit('+') | '-')
                                >> +qi::digit
                            ]
                          , qi::space
                          , integer
                          );
// here: success = false; integer = "";
```

Выключать пробелы мы научились, теперь буквально пару слов о том, как их можно включать: специально для этого есть директива <code>skip[subject]</code>, которая включает обратно автоматический парсер пробелов для парсера <code>subject</code>. Также имеется альтернативный вариант лексемы <code>skip(skip_parser)[subject]</code>, которая даёт возможность явно включить заданный парсер пробелов <code>skip_parser</code> для парсера <code>subject</code> в нужном месте.
<h3><strong>И снова к нашим баранам...</strong></h3>
А вообще, что-то я едва не забыл о нашем калькуляторе! Теперь мы достаточно много знаем об атрибутах и семантических правилах, чтобы написать вариант "интерпретирующего" калькулятора, который прямо по ходу парсинга будет выполнять необходимые арифметические операции.

К слову, в начале этой части я говорил о такой записи: <code>qi::_val = qi::_1</code>, так вот существует возможность использовать не только operator=, также можно пользоваться целым перечнем операторов присваивания <code>+=, -=, *=, /=, %=</code> и так далее, что сделает нашу работу ещё проще. Теперь это просто как два пальца!

Грамматика для интерпретатора:

```cpp
#ifndef __CALCULATOR_GRAMMAR_INTERPRETER_HPP__
#define __CALCULATOR_GRAMMAR_INTERPRETER_HPP__

#include <boost/spirit/include/qi.hpp>

namespace qi     = boost::spirit::qi;
namespace spirit = boost::spirit;

template <typename Iterator>
struct calculator_interpreter
    : qi::grammar<Iterator, int(), qi::space_type>
    {
    calculator_interpreter()
        : calculator_interpreter::base_type(expr)
        {
        expr =
            term                    [ qi::_val =    qi::_1 ]
            >> *(   '+' >> term     [ qi::_val +=   qi::_1 ]
                |   '-' >> term     [ qi::_val -=   qi::_1 ]
                )
            ;

        term =
            factor                  [ qi::_val =    qi::_1 ]
            >> *(   '*' >> factor   [ qi::_val *=   qi::_1 ]
                |   '/' >> factor   [ qi::_val /=   qi::_1 ]
                )
            ;

        factor =
                qi::uint_           [ qi::_val =    qi::_1 ]
            |   '(' >> expr         [ qi::_val =    qi::_1 ] >> ')'
            |   '-' >> factor       [ qi::_val =   -qi::_1 ]
            |   '+' >> factor       [ qi::_val =    qi::_1 ]
            ;

        }

    qi::rule<Iterator, int(), qi::space_type>
           expr, term, factor;
    };

#endif
```

Написание семантических правил не составило никакой трудности, коль скоро мы понимаем, что <code>qi::_val</code> — это то число, что получится в результате данной операции, а <code>qi::_1</code> — аргумент для этой операции, над которым мы выполняем необходимое арифметическое действие. В качестве парсера пробелов мы оставляем стандартный <code>qi::space_type</code> парсер.

Ну и совсем простой код для тестирования данной грамматики, почти ничем не отличающийся от того, что мы писали ещё в первой части:

```cpp
#include <iostream>
#include <string>

#include <boost/spirit/include/qi.hpp>

#include "calculator_grammar_interpreter.hpp"

int main()
    {
    std::cout << "//////////////////////////////////////////////\n\n";
    std::cout << "Expression parser...\n";
    std::cout << "//////////////////////////////////////////////\n\n";
    std::cout << "Type an expression... or [q or Q] to quit\n\n";

    std::string expression;

    calculator_interpreter<std::string::iterator> calc;

    while(true)
        {
        std::getline(std::cin, expression);
        if(expression == "q" || expression == "Q") break;
        std::string::iterator  begin = expression.begin()
                             , end   = expression.end();

        int result;
        bool success = qi::phrase_parse( begin
                                       , end
                                       , calc
                                       , qi::space
                                       , result);

        std::cout << "-----------------------\n";
        if(success && begin == end)
            {
            std::cout << "Parsing succeeded\n";
            std::cout << "result = " << result << "\n";
            }
        else
            {
            std::cout << "Parsing failed\nstopped at: \""
                      << std::string(begin,end) << "\"\n";
            }
        std::cout << "-----------------------\n";
        }
    }
```

Отлично! Вот и считать научились, очень похвально. Но опять же: вряд ли обычным калькулятором кого-то возможно в наше время удивить, да и сделать много полезного тоже сложно, ибо есть множество профессиональных и не очень инструментов, которые запросто заткнут нашу игрушку за пояс, так что не будем особо радоваться по этому поводу, а посмотрим, что можно сделать дальше.

Хорошо было бы, если бы мы могли, например, переводить наше выражение в набор промежуточных команд, чтобы исполнять их где-нибудь на другом устройстве, которое будет поддерживать нашу систему команд, ну или чтобы оптимизации там какие можно было провести потом. Также было бы неплохо составлять хотя бы минимальные сообщения об ошибках в записи выражения, это могло бы сэкономить кучу времени, если записи будут очень длинными.

Вот созданием такого калькулятора-компилятора мы и займёмся в следующий раз.

Заодно наконец познакомимся с тем, как можно делать в семантических действиях больше полезного и интересного с меньшими физическими и психическими затратами на примере инструментария, что предоставляет нам изрядно уже прожужжавший все уши <code>boost::phoenix</code>.
